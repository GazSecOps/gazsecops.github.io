Engineering manager: We’ve got security as code! Look, the dashboard shows our scans running in CI.

Me: That’s scanning as code. Not the same thing.

Engineering manager: What’s the difference?

Me: About five years of security incidents you haven’t had yet.

"Adding a scanner to your pipeline is like adding a smoke detector to a building. Useful, but not the same as building with fire-resistant materials in the first place."

Security as Code isn’t about running more scanners automatically. It’s about fundamentally changing how security gets built into systems. Most organizations miss this. They automate detection, declare victory, then wonder why breaches still happen.

The Misunderstanding#

I’ve watched this pattern repeat across dozens of organizations:

  1. Management reads about DevSecOps and Security as Code
  2. Security team gets budget for scanning tools
  3. Tools get integrated into CI/CD pipelines
  4. Dashboard shows lots of scans running
  5. Everyone declares success
  6. Systems still get compromised

The problem? They automated detection, not security.

What they think Security as Code means:

  • Run Snyk in CI/CD: ✓
  • Run SAST scanner: ✓
  • Run container scanner: ✓
  • Dashboard shows “90% pass rate”: ✓

What Security as Code actually means:

  • Security policies expressed as executable code
  • Infrastructure that can’t be misconfigured
  • Applications that fail to compile if insecure
  • Security requirements tested like functional requirements
  • Secure defaults that developers actually use

See the difference? One is detection. The other is construction.

The Real Problem (It’s Not Technical)#

Early in my career, I spent three weeks getting a firewall change approved. Not because the change was complex. Because the process was designed around mistrust.

Me: I need ports 8080-8090 opened between app and database servers.

Security: Submit a change request with business justification.

Me: Done.

Security: This needs architecture review.

Me: For opening ports between two internal servers?

Security: Process is process.

Three weeks later, change approved. Implemented in 30 seconds.

The problem wasn’t technical. We had the capability to make the change instantly. The problem was organizational. Security operated as a gatekeeper, not an enabler.

Security as Code isn’t primarily about technology. It’s about shifting from control to enablement. From gates to guardrails.

What It Actually Looks Like#

Approach 1: Scanning (What Most Do)#

# .gitlab-ci.yml
security_scan:
  stage: test
  script:
    - snyk test
    - trivy scan image:latest
    - semgrep --config=auto
  allow_failure: true  # Because it breaks too often

This catches known vulnerabilities. Useful. But it doesn’t prevent insecure design. Doesn’t enforce security architecture. Doesn’t make secure development the default path.

Approach 2: Security as Code (What Few Do)#

Terraform defines what you want:

resource "aws_s3_bucket" "data" {
  bucket = "company-data"
}

resource "aws_s3_bucket_public_access_block" "data" {
  bucket                  = aws_s3_bucket.data.id
  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

resource "aws_s3_bucket_server_side_encryption_configuration" "data" {
  bucket = aws_s3_bucket.data.id

  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "AES256"
    }
  }
}

Then you enforce it with a policy test that runs in CI.

Example OPA policy (Conftest) that fails if a bucket is missing encryption or public access block:

package main

buckets[b] {
  rc := input.resource_changes[_]
  rc.type == "aws_s3_bucket"
  after := rc.change.after
  after != null
  b := after.bucket
}

sse_buckets[b] {
  rc := input.resource_changes[_]
  rc.type == "aws_s3_bucket_server_side_encryption_configuration"
  after := rc.change.after
  after != null
  b := after.bucket
}

pab_buckets[b] {
  rc := input.resource_changes[_]
  rc.type == "aws_s3_bucket_public_access_block"
  after := rc.change.after
  after != null
  b := after.bucket
}

deny[msg] {
  b := buckets[_]
  not sse_buckets[b]
  msg := sprintf("S3 bucket '%s' missing default encryption configuration", [b])
}

deny[msg] {
  b := buckets[_]
  not pab_buckets[b]
  msg := sprintf("S3 bucket '%s' missing public access block", [b])
}

deny[msg] {
  rc := input.resource_changes[_]
  rc.type == "aws_s3_bucket_public_access_block"
  after := rc.change.after
  after != null
  after.block_public_policy != true
  msg := sprintf("S3 public access block allows public policy (bucket=%v)", [after.bucket])
}

And the CI commands:

terraform init
terraform plan -out plan.out
terraform show -json plan.out > plan.json

conftest test plan.json -p policy/

Now security is enforced at infrastructure creation time. Developer can’t create an insecure bucket even if they try. Security properties tested like functional properties.

That’s the difference.

The Three Shifts#

Real Security as Code requires three fundamental changes:

Shift 1: From Detection to Prevention#

Old way:

  1. Developer writes code
  2. Code gets deployed
  3. Scanner finds vulnerability
  4. Create ticket
  5. Developer fixes
  6. Repeat

New way:

  1. Developer writes code
  2. Security tests run automatically
  3. Build fails if insecure
  4. Developer fixes before merge
  5. Only secure code gets deployed

Prevention is cheaper than detection. Always.

Shift 2: From Manual Review to Automated Enforcement#

Old way: Security team reviews every change. Becomes bottleneck. Developers wait. Security becomes “the team that slows us down.”

New way: Security requirements expressed as code. Automated checks enforce them. Security team builds tools, not reviews changes.

I’ve watched security teams go from reviewing 50 changes per week (badly, because overwhelmed) to enforcing security on 500 changes per day (consistently, because automated).

Shift 3: From Compliance to Engineering#

Old way: Security is a compliance function. Write policies. Check boxes. Generate reports for auditors.

New way: Security is an engineering function. Build libraries. Create templates. Write tests. Improve continuously.

At one company, security team stopped writing policies and started writing code. Within six months, security defects dropped 70%. Policies didn’t change. Implementation did.

The Hard Parts Nobody Talks About#

Hard Part 1: Security People Don’t Want This#

Many security professionals chose security because they liked being gatekeepers. They enjoyed reviewing changes, having approval authority, setting policies.

Security as Code removes most of that. Instead of reviewing, you’re building. Instead of approving, you’re enabling. Instead of writing policies, you’re writing tests.

Some security people hate this. They leave. That’s fine - security as code isn’t for everyone.

But organizations need to understand: implementing security as code will cost you some security team members who can’t or won’t make the transition.

Hard Part 2: It’s Slower at First#

Automating security properly takes time upfront. You can buy a scanner and get results tomorrow. Building proper security automation takes months.

Management: How long to implement security as code?

Me: Six months for basic framework. Year for broad coverage.

Management: We can buy a scanner for next quarter.

Me: Scanner will find vulnerabilities. Won’t prevent them. Won’t scale. Won’t change culture.

Management: But we’ll have metrics for the board meeting.

Most organizations take the scanner. Get quick metrics. Miss the fundamental transformation.

Hard Part 3: Perfect is the Enemy of Done#

Security teams try to automate everything perfectly. Create checks for everything. Cover every edge case. Build the ultimate security framework.

Meanwhile, developers wait. Get frustrated. Route around security. Security framework never ships.

Better approach: Start small. Automate the highest-impact checks first:

  • Hardcoded credentials
  • Unencrypted data
  • Excessive permissions
  • Known CVEs in dependencies

Ship that. Show value. Build trust. Then expand.

I’ve never seen security automation succeed when implemented all at once. I’ve seen incremental approaches work repeatedly.

Practical Examples#

Example 1: Infrastructure That Can’t Be Misconfigured#

This is what “guardrails” looks like in the real world: modules and policies.

Modules give you secure defaults. Policies stop people “just this once”-ing their way into a breach.

Example: an internal Terraform module that doesn’t offer a “turn off encryption” variable.

# app/main.tf
module "secure_bucket" {
  source = "./modules/secure_s3"
  name   = "company-data"
}

And modules/secure_s3 always creates the public access block and default encryption.

Then your Conftest policy (like the one above) stops anyone bypassing the module.

Example 2: Security Tests in CI#

If you want build-breaking checks, use tools that already exist and wire them in so they fail fast.

Examples that work well in practice:

# Stop secrets getting committed
gitleaks detect --source .

# Alternative secret scanner (also good)
trufflehog filesystem .

# Stop known-vulnerable dependencies
osv-scanner --recursive .

# Stop insecure infrastructure changes
terraform plan -out plan.out
terraform show -json plan.out > plan.json
conftest test plan.json -p policy/

Security requirements tested like functional requirements. Build fails if violated. No manual review needed.

Example 3: Secure Defaults#

Secure defaults means developers get the safe thing without thinking.

Example: set sane cookie flags and headers in the one shared place every service uses.

http.SetCookie(w, &http.Cookie{
    Name:     "session",
    Value:    sessionID,
    Secure:   true,
    HttpOnly: true,
    SameSite: http.SameSiteStrictMode,
})

w.Header().Set("Content-Security-Policy", "default-src 'self'")
w.Header().Set("X-Content-Type-Options", "nosniff")
w.Header().Set("X-Frame-Options", "DENY")

The point isn’t Go. The point is that it lives in a shared library or template, not a wiki page.

Secure by default. Developer gets security for free. Insecurity requires effort (and explanation).

The Cultural Shift#

Technology is the easy part. Culture is hard.

What Needs to Change#

Security team mindset:

  • From: “We review and approve changes”
  • To: “We build tools that enable secure development”

Developer mindset:

  • From: “Security will check this later”
  • To: “Security is my responsibility”

Management mindset:

  • From: “Security is a compliance function”
  • To: “Security is an engineering discipline”

What This Looks Like in Practice#

At a financial services company, we made this shift:

Before:

  • Security reviewed every change manually
  • Average review time: 3 days
  • Developer satisfaction with security: 2/10
  • Security defects per quarter: ~150

After:

  • Security requirements automated
  • Average “review” time: 5 minutes (automated)
  • Developer satisfaction: 8/10
  • Security defects per quarter: ~20

Security team went from 12 people (reviewing changes) to 4 people (building automation). Developers were happier. Security was better. Cost was lower.

But getting there required:

  • 8 security team members leaving (couldn’t adapt)
  • 6 months of building automation
  • Management accepting initial slowdown
  • Developers learning to write security tests

Not easy. But worth it.

Common Failures#

Failure 1: Tool-First Approach#

Management: We bought a DevSecOps platform! We have security as code now!

Me: What changed in your development process?

Management: We run more scans now.

Me: That’s not security as code. That’s scanning as code.

Management: But the vendor said-

Me: The vendor wants to sell tools. You need to change culture.

Tools don’t create security as code. Culture creates security as code. Tools support it.

Failure 2: Trying to Automate Everything#

Security team tries to build automation covering every possible security issue. Takes two years. Never ships. Developers route around it.

Better: Automate 20% of issues that cause 80% of problems. Ship in 3 months. Show value. Build trust. Iterate.

Failure 3: No Developer Buy-In#

Security team builds automation without consulting developers. Tools are slow, noisy, painful. Developers disable them or work around them.

Better: Build automation with developers. Make it fast, quiet, helpful. Developers will actually use it.

Failure 4: Treating Symptoms, Not Causes#

Organization has lots of SQL injection vulnerabilities. Security team adds SQL injection scanner to pipeline.

Problem: Scanner finds vulnerabilities after code is written. Doesn’t prevent them.

Better: Provide secure database library that prevents SQL injection by default. Make secure code easier than insecure code.

What Success Looks Like#

After implementing security as code properly, you should see:

Metrics:

  • Security defects drop 70-90%
  • Time to deploy decreases
  • Developer satisfaction with security increases
  • Security team size stays flat or decreases while coverage increases

Behaviors:

  • Developers write security tests without being asked
  • Infrastructure can’t be created in insecure configurations
  • Security issues caught before code review
  • Security team focused on building tools, not reviewing changes

Culture:

  • Security seen as enabler, not blocker
  • “Secure by default” becomes reality
  • Security is everyone’s responsibility
  • Continuous improvement in security automation

The Hard Truth#

Security as code requires fundamental changes:

  1. Security team changes role - From gatekeepers to toolmakers
  2. Developers take ownership - Security becomes their responsibility
  3. Management accepts upfront cost - 6-12 months before full benefits
  4. Organization tolerates turnover - Some security people won’t adapt
  5. Culture shifts - From control to enablement

Most organizations aren’t willing to make these changes. They buy scanners, declare victory, wonder why security doesn’t improve.

The few that do make these changes see dramatic improvements. But it’s hard. Takes time. Requires commitment.

Practical Steps#

If you want to actually implement security as code (not just scanning as code):

Step 1: Start Small

Pick one high-impact problem. Automate it properly. Ship it. Show value.

Examples:

  • Prevent hardcoded secrets (reject commits with secrets)
  • Enforce encryption (infrastructure can’t be created unencrypted)
  • Require authentication (builds fail if endpoints lack auth)

Step 2: Make It Fast

Security checks must be fast (<1 minute) or developers will disable them.

If your security scan takes 10 minutes, it’s too slow. Optimize or split it up.

Step 3: Make It Helpful

Security tools should help developers fix problems, not just find them.

Bad: “SQL injection vulnerability on line 47” Good: “SQL injection vulnerability on line 47. Use parameterized queries: cursor.execute('SELECT * FROM users WHERE id = ?', (user_id,))”

Step 4: Build With Developers

Don’t build security automation in isolation. Pair with developers. Get feedback. Iterate.

Step 5: Measure What Matters

Don’t measure:

  • Number of scans run
  • Number of vulnerabilities found
  • Percentage of code covered

Do measure:

  • Time from commit to production
  • Number of security defects in production
  • Developer satisfaction with security tools
  • Mean time to fix security issues

The Real Goal#

The goal of security as code isn’t to run more scanners automatically. The goal is to make insecure code impossible to deploy.

Not harder to deploy. Impossible to deploy.

When your infrastructure literally can’t be created without encryption, you don’t need to scan for unencrypted data. When your build fails if authentication is missing, you don’t need to review for missing auth.

That’s security as code. Construction, not detection.

Why Most Organizations Fail#

Honestly? Because it’s easier to buy scanners than change culture.

Scanners give you metrics quickly. Dashboards for executives. Checkboxes for auditors. No organizational change required.

Security as code requires:

  • Security team to change role (threatening)
  • Developers to take ownership (effort)
  • Management to accept delay (patience)
  • Organization to rethink security (hard)

Most organizations take the easy path. Buy scanners. Get metrics. Miss the transformation.

Then wonder why they still get breached.

A Warning#

If you’re serious about security as code, prepare for:

  1. Resistance from security team - Some will leave
  2. Initial slowdown - Building automation takes time
  3. Management impatience - They want quick wins
  4. Developer skepticism - They’ve heard this before
  5. Organizational inertia - Change is hard

But if you push through, you get:

  • Dramatically better security
  • Faster development
  • Happier developers
  • Smaller security team (doing more)
  • Actual security as code, not scanning as code

Worth it. But not easy.

Conclusion#

Security as code isn’t about running scanners in CI/CD. It’s about fundamentally changing how security gets built into systems.

Most organizations confuse the two. They automate detection, call it security as code, wonder why security doesn’t improve.

Real security as code means:

  • Security policies expressed as executable code
  • Infrastructure that can’t be misconfigured
  • Applications that fail to build if insecure
  • Security requirements tested like functional requirements
  • Secure defaults that developers actually use

This requires cultural change, not just tool adoption. It requires security teams to change roles, developers to take ownership, and management to invest in long-term improvement.

Most organizations won’t do this. They’ll buy scanners, get dashboards, declare success.

The few that do it properly see dramatic improvements. 70-90% reduction in security defects. Faster deployments. Happier developers. Better security.

But you have to actually do it. Not just buy tools and hope.

The scanners were never the point. The point was to fundamentally change how we build secure systems.

This post is based on 15+ years implementing security across financial services, technology companies, and startups. The patterns repeat. The failures are predictable. The successes are possible - if you’re willing to do the work.
