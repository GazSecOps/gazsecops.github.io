Password Managers in 2025: What Actually Works

Management: “We need a password manager for a team.”

Sysadmin: “Which one?”

Management: “The secure one.”

Sysadmin: “They’re all secure. What do you actually need it to do?”

Management: “…store passwords?”

This conversation happens constantly. Organizations pick password managers the same way they pick any security tool - based on vendor pitch decks and analyst reports, not actual requirements.

"Password managers solve different problems. KeePassXC for personal use, Bitwarden for teams, Vault for infrastructure. Pick based on what you're protecting and who needs access, not which one has the best marketing."

The password manager landscape in 2025 splits into three categories: personal password managers, team password managers, and infrastructure secrets management. They’re not interchangeable.

Personal Password Managers#

For individual use. Your email passwords, banking logins, personal accounts.

KeePassXC#

Local database. No cloud. You control the file.

What it does well:

  • Zero trust in cloud providers (database stays on your machines)
  • Works completely offline
  • Free and open source
  • Linux-native (actually works properly, not an afterthought)

What it doesn’t do:

  • No built-in sync (use Syncthing or Nextcloud yourself)
  • No browser autofill by default (requires setup)
  • No mobile apps from the same developers (third-party only)

When to use it:

  • You don’t trust cloud providers
  • You know how to sync files yourself
  • You want complete control

I use KeePassXC. Database syncs via Syncthing across three machines. When Syncthing breaks (it does), I fix it. When a cloud password manager has an outage, my passwords still work.

Bitwarden#

Cloud-based. Can self-host. Open source.

What it does well:

  • Works everywhere (browser extensions, mobile apps, desktop)
  • Free tier is actually usable
  • Can self-host with Vaultwarden (lighter, faster)
  • Import from basically anything

What it doesn’t do:

  • Cloud version requires trusting Bitwarden servers
  • Self-hosted requires maintaining a server
  • Mobile autofill is hit-or-miss on Linux phones

When to use it:

  • You want cloud sync that just works
  • You’re willing to trust Bitwarden (or run your own)
  • You need it to work on your mum’s phone without tech support calls

Most people should use Bitwarden. It’s the right balance of security, usability, and “just works.”

bitwarden.com

1Password#

Proprietary. Expensive. Polished.

What it does well:

  • Best UX in industry (genuinely smooth)
  • SSH key agent integration (store SSH keys, use them transparently)
  • Family/team features that don’t require a PhD
  • Secret Key protection (additional encryption layer)

What it doesn’t do:

  • Work without paying (no meaningful free tier)
  • Give you source code access
  • Let you self-host

1password.com

When to use it:

  • You’re paying anyway (company expense)
  • UX matters more than open source
  • You use SSH keys heavily

If your company pays for it, 1Password is excellent. If you’re paying yourself, Bitwarden is 90% as good for free.

Team Password Managers#

For teams sharing credentials. Support login credentials, API keys, shared accounts.

Bitwarden Organizations#

Same Bitwarden, team features enabled.

What works:

  • Collections (group passwords by project/team)
  • Granular permissions (who can see what)
  • Audit logs (who accessed what when)
  • Works like personal Bitwarden (minimal training)

What doesn’t:

  • Gets expensive for large teams
  • Permissions model is basic (not fine-grained RBAC)
  • No approval workflows

Real usage:

Engineer: I need the staging database password.

Sysadmin: Check Bitwarden, Engineering collection.

Engineer: I don’t have access.

Sysadmin: Fixed. Added you to the collection.

This workflow is fine for teams under 50 people. Beyond that, you need something with proper access control.

Passbolt#

Built for teams. Open source. Self-hosted.

What works:

  • Granular permissions (user groups, resource permissions)
  • Audit logs and compliance features
  • API for automation
  • GPG-based encryption (can use hardware keys)

What doesn’t:

  • Requires setup and maintenance
  • Browser extension can be flaky
  • Steeper learning curve than Bitwarden

passbolt.com

When to use it:

  • You need detailed access controls
  • Compliance requires self-hosted
  • You have someone to maintain it

Financial services and government contractors use Passbolt. If you need to prove who accessed what and when, Passbolt has the audit trail.

Infrastructure Secrets Management#

For machines, not humans. Database passwords, API keys, TLS certificates, cloud credentials.

HashiCorp Vault#

Industry standard. Does everything. Complex.

What it does:

  • Dynamic secrets (generate database credentials on demand)
  • Encryption as a service
  • PKI (issue and manage certificates)
  • AWS/GCP/Azure integration
  • Kubernetes secrets injection

What it costs:

  • Setup complexity (not trivial to run properly)
  • Operational overhead (Vault clusters need care and feeding)
  • License changes (newer versions have restrictive licenses)

When to use it:

  • You have infrastructure at scale
  • You need dynamic credentials
  • You have someone who knows how to run it

vaultproject.io

Example use case: Your application needs a database password. Instead of hardcoding it, the app authenticates to Vault using its Kubernetes service account. Vault generates a PostgreSQL user with a 1-hour TTL and returns the credentials. App uses them. After 1 hour, credentials expire and become useless.

This is brilliant for limiting blast radius. It’s also complicated to set up.

A minimal Vault workflow (KV secrets):

Write a secret:

vault secrets enable -path=kv kv-v2
vault kv put kv/apps/payments DB_PASSWORD='replace-me'

Read it back:

vault kv get kv/apps/payments

Lock it down with a policy:

# payments.hcl
path "kv/data/apps/payments" {
  capabilities = ["read"]
}

vault policy write payments payments.hcl

Dynamic database credentials (the bit people actually want):

This creates short-lived users instead of storing a shared password forever.

vault secrets enable database

vault write database/config/payments \
  plugin_name=postgresql-database-plugin \
  allowed_roles=payments-ro \
  connection_url='postgresql://{{username}}:{{password}}@db.internal:5432/payments?sslmode=require' \
  username='vault_admin' \
  password='vault_admin_password'

vault write database/roles/payments-ro \
  db_name=payments \
  creation_statements="CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}'; GRANT SELECT ON ALL TABLES IN SCHEMA public TO \"{{name}}\";" \
  default_ttl=1h \
  max_ttl=24h

vault read database/creds/payments-ro

The operational lesson: if you can’t rotate the underlying database admin credential safely, don’t start here. You’ll build a new dependency and then spend a year being scared of it.

OpenBao#

Community fork of Vault. Same features, different governance.

Why it exists:

HashiCorp changed Vault’s license in 2023. Open source community forked it. OpenBao is that fork.

What’s different:

  • Fully open source license (MPL 2.0)
  • Community governance
  • Feature parity with Vault OSS

What’s same:

  • API compatibility (drop-in replacement)
  • Configuration format
  • Client tools

openbao.org

When to use it:

  • You want Vault features without license concerns
  • You prefer community governance
  • You’re already using Vault and want to avoid license issues

If you’re starting fresh and want Vault-like features, OpenBao is the safer bet long-term.

In practice:

  • Vault CLI is vault
  • OpenBao CLI is bao

Most examples port over directly.

Cloud Provider Secrets Managers#

AWS Secrets Manager, Azure Key Vault, Google Secret Manager

Advantages:

  • Integrated with cloud IAM (permissions work like everything else)
  • No servers to maintain

Disadvantages:

  • Vendor lock-in (hard to migrate away)
  • Less powerful than Vault (no dynamic secrets, limited PKI)
  • Costs add up at scale

When to use:

  • You’re all-in on one cloud provider
  • You need something simple that works
  • You don’t want to run Vault

Most startups should start here. If you’re on AWS anyway, Secrets Manager is fine. When you outgrow it, migrate to Vault/OpenBao.

Useful commands (so you can stop clicking around portals):

AWS Secrets Manager:

aws secretsmanager create-secret \
  --name payments/db_password \
  --secret-string 'replace-me'

aws secretsmanager get-secret-value \
  --secret-id payments/db_password \
  --query SecretString \
  --output text

Azure Key Vault:

az keyvault secret set \
  --vault-name kv-prod \
  --name payments-db-password \
  --value 'replace-me'

az keyvault secret show \
  --vault-name kv-prod \
  --name payments-db-password \
  --query value \
  -o tsv

Google Secret Manager:

printf '%s' 'replace-me' | gcloud secrets create payments-db-password --data-file=-
gcloud secrets versions access latest --secret=payments-db-password

Hardware-Backed Security#

For when software encryption isn’t enough.

TPM (Trusted Platform Module)#

Built into most modern hardware. Stores encryption keys in hardware.

What it’s used for:

  • LUKS disk encryption on Linux
  • Measured boot (verify system hasn’t been tampered with)
  • Storing credentials safely

Limitations:

  • Tied to specific hardware
  • Not portable across machines
  • Complex to work with directly

You’re probably already using TPM without knowing it. If you enabled full disk encryption on a modern laptop, TPM is storing the key.

YubiHSM / Nitrokey HSM#

Small, affordable hardware security modules.

Use cases:

  • Code signing keys (sign releases without key leaving HSM)
  • Certificate authority (CA private key never on disk)
  • Air-gapped environments

Example setup:

Certificate authority private key lives on YubiHSM. When you need to sign a certificate, you authenticate to the HSM. HSM performs signing operation. Private key never leaves hardware.

If someone compromises the server, they get nothing - they can’t extract the key.

If you’re going down this road, you’ll be living with PKCS#11.

Basic sanity checks (generic PKCS#11):

# List slots and tokens
pkcs11-tool --module /path/to/your/pkcs11.so --list-slots

# List objects on a token
pkcs11-tool --module /path/to/your/pkcs11.so --login --list-objects

Then you configure your CA / signing tooling to use the PKCS#11 key handle instead of a key file on disk.

Reality check: the HSM isn’t the hard part. The hard part is provisioning, backups, PIN management, and not bricking the thing during a rushed “quick change”.

Reality check:

Most organizations don’t need HSMs. They need to stop putting passwords in git repositories.

HSMs solve a real problem (protecting private keys), but that’s not most people’s problem. Most breaches happen because someone committed AWS credentials to a public repo.

What You Actually Need#

For personal use: KeePassXC or Bitwarden.

For a small team (under 20): Bitwarden Organizations.

For a larger team (20-100): Bitwarden Organizations or Passbolt (if compliance matters).

For infrastructure secrets: Start with cloud provider secrets manager. Move to Vault/OpenBao when you need dynamic credentials.

For signing keys / CA keys: YubiHSM if you’re serious about security. Software keys are probably fine otherwise.

Migration Tips#

LastPass to Bitwarden: Export from LastPass as CSV. Import to Bitwarden. Delete LastPass account. Takes 10 minutes.

KeePass to Bitwarden: Export KeePass database as CSV. Import to Bitwarden. Keep KeePass database as backup. Sync issues go away.

No password manager to Bitwarden: Install browser extension. Save passwords as you log in. Takes a month to capture everything. Worth it.

Hardcoded secrets to Vault: This is the painful one. Requires code changes. Do it gradually:

  1. Identify all hardcoded secrets
  2. Move to environment variables first (intermediate step)
  3. Migrate environment variables to Vault
  4. Update apps to fetch from Vault

Don’t try to do it all at once. One service at a time.

Common Mistakes#

Storing infrastructure secrets in team password manager:

Don’t put your production database password in Bitwarden alongside the marketing team’s social media logins.

Humans access Bitwarden. Machines access Vault. Keep them separate.

Not rotating credentials after migration:

Migrated from LastPass to Bitwarden? Great. Did you change all the passwords?

No? Someone with access to the old export can still log in.

Rotation is boring. It’s also necessary.

Over-engineering personal password management:

You don’t need Vault for your Netflix password. You need KeePassXC or Bitwarden.

Vault is for infrastructure. It’s overkill for personal use.

Under-engineering infrastructure secrets:

Storing production database passwords in a text file on the server is not a secrets management strategy.

If your app can read the file, so can an attacker who compromises the app.

The Boring Truth#

Most password security problems aren’t technology problems. They’re process problems.

Real issues:

  • Passwords shared via Slack DMs
  • Sticky notes on monitors
  • Same password for everything
  • Never rotating credentials
  • Not using a password manager at all

Not real issues:

  • Which password manager you picked
  • Self-hosted vs cloud
  • Open source vs proprietary

Pick any decent password manager. Use it. That’s 90% of the battle.

The remaining 10% is operational discipline. Rotate credentials. Don’t share passwords in Slack. Use unique passwords. Enable 2FA.

What’s Coming#

Passkeys:

FIDO2/WebAuthn passkeys are finally getting traction. 1Password, Bitwarden, and others can sync them across devices.

They’re good because they remove the password from the phishing equation. A passkey won’t sign in to evil-example.com because the origin doesn’t match.

They still aren’t a full replacement for passwords yet:

  • You still need recovery flows (people lose phones)
  • Many services still fall back to “email link” or SMS (which attackers love)
  • Enterprises still need lifecycle management (joiners/movers/leavers)

So yes, use passkeys where you can. You still need a password manager because you’ll be living in the hybrid world for years.

Automated rotation:

Vault and cloud providers are getting better at automatic credential rotation.

Instead of rotating database passwords manually every 90 days, Vault rotates them automatically every 24 hours.

This is the future. Also requires apps that handle credential rotation gracefully (most don’t).

Better integration:

Password managers are integrating with SSH, git, and other tools.

1Password SSH agent: Store SSH keys in 1Password, use them transparently. No keys on disk.

This is excellent for security. It’s also new enough that it breaks occasionally.

Bottom Line#

For most people: Bitwarden (free, works everywhere, open source).

For infrastructure: Cloud provider secrets manager to start, Vault/OpenBao when you need dynamic credentials.

For personal use if you don’t trust clouds: KeePassXC (local, you control everything).

For teams with compliance requirements: Passbolt (self-hosted, detailed audit logs).

For signing/CA keys: YubiHSM if you’re serious, software keys otherwise.

Don’t overthink it. Pick one. Use it. That’s infinitely better than password reuse or sticky notes.
