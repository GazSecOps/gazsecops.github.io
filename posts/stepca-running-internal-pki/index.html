Smallstep step-ca: Running Internal PKI Without Losing Your Mind

If you read my post on building your own CA and thought “fine, but how do I actually issue certs without becoming the Certificate Person”, this is the next step.

step-ca (Smallstep Certificate Authority) is the bit that makes internal PKI usable: automated issuance, short-lived certs, predictable renewal, and enough policy to stop someone minting a wildcard for your entire estate.

It also gives you a new thing to break at 3am.

On-call: Everything that talks TLS is failing.

Engineer: But we rotated the certs yesterday.

Sysadmin: You rotated them to a CA chain nobody trusts. Also the CA is down.

Management: How can the CA being down break existing connections?

Sysadmin: It doesn’t. The expiry storm you scheduled does.

Internal PKI fails in three ways: you lose the keys, you lose the trust, or you lose control of issuance.

This post covers a practical step-ca setup on a Linux VM, what to lock down, how to issue and renew certs, and the failure modes I’ve seen in the wild.

What step-ca Actually Buys You#

You can run an internal CA with OpenSSL and a folder of files. People do. It works right up until you have:

  • 400 services needing certs
  • 6 different teams doing it “their way”
  • one critical service with a cert that expires on Christmas Day

step-ca buys you boring consistency:

  • one place to issue certs
  • one policy layer (who can get what, with what SANs)
  • automation via ACME and step CLI
  • short lifetimes so stolen certs are less useful

If you haven’t read the CA post, start there for the threat model and the “don’t leak the root key” rules.

Architecture: Keep the Root Offline#

The safe default:

  • Root CA key: offline, rarely used
  • Intermediate CA key: online, used for issuing
  • step-ca: runs with the intermediate key

That way:

  • compromise of the step-ca box does not give an attacker the root key
  • you can rotate the intermediate without redeploying trust roots everywhere

If you put the root key on the same VM as your issuing CA, you’re not running PKI. You’re running “one box to steal”.

A Practical Linux VM Setup (Systemd)#

This is the simplest deployment that still holds up:

  • a dedicated Linux VM
  • step-ca running as its own user
  • a firewall rule that only allows traffic from where it should be used
  • backups of CA state (encrypted, tested restore)

System hardening (minimum viable)#

  • don’t run it as root
  • don’t expose the admin API to the internet
  • log everything you can, because you will need it later

Example systemd unit (trimmed for sanity):

[Unit]
Description=step-ca
After=network.target

[Service]
User=step
Group=step
ExecStart=/usr/bin/step-ca /etc/step-ca/config/ca.json
Restart=on-failure
NoNewPrivileges=true
PrivateTmp=true
ProtectSystem=strict
ProtectHome=true
ReadWritePaths=/etc/step-ca /var/lib/step-ca

[Install]
WantedBy=multi-user.target

The point is not to win a hardening contest. It’s to reduce the blast radius when someone inevitably gets a foothold somewhere.

Network rules (the bit everyone forgets)#

If every developer laptop can hit your CA, someone will eventually script issuance until you notice your logs are on fire.

Lock it down:

  • only allow traffic from your networks and clusters
  • if you have multiple environments, separate them (dev CA vs prod CA)

Yes, this is annoying. It’s still less annoying than revoking half your estate because someone minted certs for *.prod.internal.

Provisioners: How Do You Decide Who Gets a Cert?#

This is where most internal PKI goes wrong.

If your rule is “anyone who can reach the CA can get a cert”, you’ve built a certificate vending machine.

Provisioners are the authentication and policy layer. Pick one based on how your services authenticate.

Common patterns:

1. ACME (good default for services)#

ACME works well for:

  • services that can do automated renewal
  • environments that already understand Let’s Encrypt style flows

The two big decisions are:

  • HTTP-01 vs DNS-01 (internal DNS is usually messy)
  • what names are allowed (policy matters more than the challenge)

2. OIDC / JWT (good for workloads with identity)#

If your workloads already have identity tokens (cloud identity, workload identity, service accounts), use that. It gives you:

  • traceability (who requested what)
  • policy based on claims

3. mTLS provisioners (good for controlled environments)#

If you already have a bootstrap trust (for example, a machine identity you trust), you can use mTLS to request new certs.

It works well for fleets. It also fails badly if your bootstrap identity leaks.

Pick one. Don’t enable everything “just in case”. That’s how you end up with three bypass paths nobody remembers.

Issuance Policy: Stop Wildcards and Other Regrets#

You need to decide what you will issue. Write it down and enforce it.

Reasonable defaults:

  • no wildcards in production
  • short lifetimes (hours or days, not months)
  • SANs must be within approved internal zones
  • different provisioners for dev vs prod

Example policy choices that save you later:

  • issue separate intermediates per environment (dev, staging, prod)
  • keep names boring (svc-name.namespace.svc.cluster.local, service.prod.internal)
  • ban random personal vanity names (they will end up in production somehow)

Renewal: The Bit That Makes Or Breaks This#

If renewal is not automatic, the CA will become a calendar reminder system.

For Linux services, the most boring approach is usually best:

  • a small renewal script
  • a systemd timer
  • reload the service when the cert rotates

Example shape:

step ca renew /etc/tls/service.crt /etc/tls/service.key \
  --ca-url https://ca.prod.internal:443 \
  --root /etc/step-ca/certs/root_ca.crt

systemctl reload your-service

The important part is not the command. It’s the behaviour:

  • renew before expiry
  • alert when renewal fails
  • don’t schedule every cert to renew at the same time

Expiry storms are self-inflicted outages. Spread renewal. Use jitter. Your future self will thank you.

Backups and Recovery (Because This Will Go Wrong)#

You need to answer one question:

“If this VM dies, how do we keep issuing certs?”

Back up:

  • step-ca state (database)
  • configuration
  • issuing keys (intermediate) and password material (securely)
  • the root and intermediate certificates (public)

Rules:

  • encrypt backups
  • restrict access (least privilege)
  • test restore, not just backup

If you can’t restore, you don’t have backups. You have feelings.

Monitoring: The Few Things Worth Alerting On#

You don’t need 40 dashboards. You need a few signals:

  • issuance rate (spikes can indicate abuse)
  • renewal failure rate
  • CA availability and latency
  • certificate expiry in critical systems (external check)

And you want a list of:

  • which services have certs from which intermediate
  • where trust roots are deployed

When you rotate intermediates, this is the difference between a controlled change and a two-week scavenger hunt.

Concrete Prometheus Setup#

Export metrics from step-ca via the /metrics endpoint (enable in config). Then alert on:

groups:
  - name: step-ca
    rules:
      - alert: StepCADown
        expr: up{job="step-ca"} == 0
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "step-ca is down"

      - alert: StepCACertificateIssuanceSpike
        expr: |
          rate(step_ca_certificates_issued_total[5m]) 
          > 10 * (rate(step_ca_certificates_issued_total[5m] offset 1h))
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Unusual certificate issuance rate"

      - alert: StepCACertificateRenewalFailures
        expr: rate(step_ca_renewal_failures_total[5m]) > 0
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Certificate renewals failing"

Quick CLI check:

curl -s http://ca.prod.internal:9000/metrics | grep step_ca
curl -s http://ca.prod.internal:9000/health

What Actually Breaks#

1. Trust distribution#

Issuing certs is the easy part. Getting everything to trust your CA chain is the work.

You will find:

  • containers with their own CA bundles
  • Java apps with a private trust store nobody remembers
  • appliances that can’t be updated without a support ticket

Write down how trust gets deployed for:

  • Linux hosts (/usr/local/share/ca-certificates, whatever your distro uses)
  • containers (bake into base images, don’t hotfix)
  • JVM apps (decide a pattern and enforce it)

2. Names and SAN sprawl#

Someone will request a cert with 14 SANs because “it was easier”. That cert will leak somewhere. Now you have a single private key that unlocks half your estate.

Keep SANs tight. If something needs multiple names, ask why.

3. DNS and split-horizon surprises#

ACME challenges and service identity both depend on DNS. If your internal DNS is split across clouds, VPNs, and cluster DNS, this becomes a steady source of pain.

If you want a dedicated DNS post: DNS Security: What Actually Breaks.

4. “We migrated the CA” outages#

This is the classic:

  • new CA deployed
  • new intermediate created
  • half the services start issuing from the new chain
  • trust roots not deployed everywhere
  • some clients now see an untrusted chain and fail hard

Fix:

  • deploy trust first
  • then switch issuance
  • keep the old intermediate around long enough for existing certs to age out

Don’t flip everything in one day unless you like incident bridges.

Key Storage: The Hard Truth#

If you can, use hardware-backed keys for the intermediate.

If you can’t, at least:

  • lock down filesystem permissions
  • separate the CA VM from general workloads
  • avoid putting key material in build pipelines

The intermediate key is the crown jewels for issuance. Treat it that way.

The Boring Truth#

step-ca is not magic. It’s a decent tool that makes internal PKI survivable.

If you want it to stay survivable:

  • keep the root offline
  • lock down who can request what
  • make renewal boring and automatic
  • back it up and test restore
  • plan trust distribution like it’s a deployment, because it is

If you do those, internal PKI becomes a background service. Quiet. Predictable. Slightly irritating. Which is exactly how you want it.
