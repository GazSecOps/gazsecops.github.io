Management: We need a penetration test for compliance.

Sysadmin: What are we testing?

Management: Everything.

Sysadmin: That’s not a scope. What’s the goal?

Management: The goal is to check the box that says we had a penetration test.

Sysadmin: Then you don’t need a pentest. You need a rubber stamp.

Most penetration testing is security theater. A PDF appears once a year, lists findings you already knew about, gets filed away, nothing changes. The checkbox is ticked. Compliance is satisfied. Security hasn’t improved.

A pentest that tells you what you already know is an expensive way to avoid fixing problems. A pentest that surprises you is valuable - but only if you actually fix what it finds.

This isn’t about shitting on pentesters. Good pentesters are worth their weight in gold. This is about how to get actual value from penetration testing instead of buying an expensive PDF that nobody reads.

What Penetration Testing Actually Tells You#

A pentest answers one specific question: at this moment, could a skilled attacker with this scope compromise this target?

It does not tell you:

  • Whether you’re “secure” (security isn’t binary)
  • What vulnerabilities will exist next month
  • Whether your developers write secure code
  • Whether your security team is competent
  • Whether you’ll pass your audit

It does tell you:

  • What an attacker can reach from a given starting point
  • Which controls actually stop attacks vs look good on paper
  • How far an attacker could get before detection
  • Whether your incident response works when someone’s actually attacking

The value isn’t in the PDF. The value is in what you learn and what you change.

The Problem With Most Pentests#

Problem 1: Compliance-driven scope#

“We need a pentest for PCI/SOC2/ISO27001.”

Scope becomes: whatever the auditor asks for. Usually external perimeter, maybe internal network if you’re lucky. Time-boxed to whatever the budget allows.

Result: pentester scans your public IP range, finds the same vulnerabilities your vulnerability scanner found, writes a report, gets paid. You already knew about those vulnerabilities. Nothing new learned.

Problem 2: Fixed time, fixed price#

Pentest scoped for 5 days. Pentester finds critical issue on day 1. What happens?

Option A: Pentester stops, reports critical, engagement ends. You got lucky - critical found early.

Option B: Pentester continues exploring because they’re paid for 5 days. Finds more issues. Good, but critical wasn’t reported early.

Option C: Pentester runs out of time before exploring everything. Scope was too big for the time. Findings are incomplete.

None of these are ideal. Pentesting shouldn’t be time-boxed. It should be scope-boxed with a time estimate.

Problem 3: Testing production only#

Pentest targets production. Findings are: critical vulnerabilities in production.

Now you have a problem: you can’t fix them without a production change, production changes require testing, testing can’t replicate production exactly, so you ship fixes to production and hope.

Better approach: test pre-production environments. Find issues before they hit production. But this requires pre-production environments that actually match production, which many organisations don’t have.

Problem 4: No retesting#

Pentest finds 10 issues. You fix 9 of them. One fix was wrong. How do you know?

Without retesting, you don’t. The pentest report said “fix this,” you fixed something, now you assume it’s fixed.

Retesting should be part of every pentest engagement. Not a separate line item. Part of the deliverable.

Problem 5: Findings that can’t be fixed#

Pentest finds vulnerability in legacy system that can’t be patched without replacing the entire system. Cost to fix: millions. Risk of exploitation: unclear.

This isn’t actionable. The finding exists, but you can’t do anything with it.

Better approach: pentester works with you to understand constraints. Finding becomes “legacy system has known vulnerability, recommend compensating controls: network segmentation, monitoring, migration plan.”

How to Scope a Pentest That Actually Provides Value#

Step 1: Define what you’re worried about#

Not “everything.” Specific concerns:

  • “We’re worried about ransomware spreading from our office network”
  • “We’re worried about someone stealing customer data from our web application”
  • “We’re worried about a compromised developer credential leading to production access”
  • “We’re worried about our cloud infrastructure being misconfigured”

Each concern maps to a different scope.

Step 2: Define the starting point#

Where does the attacker start?

External: Internet-facing systems, no credentials Internal: Inside the network, maybe with domain user credentials Cloud: With read-only cloud console access Physical: On-premises, trying to plug into network ports Social engineering: Targeting employees via phishing, vishing, pretexting Supply chain: Starting from a vendor or partner access point

Different starting points test different controls. External tests your perimeter. Internal tests your segmentation and endpoint security. Cloud tests your IAM and configuration.

Step 3: Define the goal#

What counts as success for the attacker?

  • “Access to customer database”
  • “Domain administrator access”
  • “Ability to exfiltrate data”
  • “Access to production Kubernetes cluster”
  • “Code execution on production servers”

Clear goals make for clear findings. “Attacker could have achieved X by exploiting Y” is actionable. “Attacker found vulnerabilities” is not.

Step 4: Define what’s in scope and out of scope#

In scope:

  • Specific IP ranges
  • Specific applications
  • Specific cloud accounts
  • Specific attack vectors

Out of scope:

  • Denial of service (unless specifically agreed)
  • Social engineering of specific individuals (unless consented)
  • Physical security (unless specifically scoped)
  • Third-party systems you don’t control

Be specific. “All cloud infrastructure” is too vague. “AWS account 123456789012, production VPC, EKS cluster and associated services” is specific.

Sample Scope Document#

# Penetration Test Scope

## Engagement Details
- **Client:** Acme Corp
- **Duration:** 10 business days (excluding retesting)
- **Testing window:** 09:00-18:00 UTC, Monday-Friday
- **Start date:** 2025-03-10
- **Primary contact:** Jane Smith (jane@acme.corp)
- **Emergency contact:** +44 7XXX XXXXXX

## Objectives
1. Assess external perimeter security
2. Attempt to access internal network from external starting point
3. Evaluate detection and response capabilities

## Scope

### In Scope
**IP Ranges:**
- 203.0.113.0/24 (public-facing services)
- 198.51.100.10-20 (API endpoints)

**Applications:**
- https://app.acme.corp (customer portal)
- https://api.acme.corp (public API)
- https://admin.acme.corp (internal admin, accessible via VPN)

**Cloud Infrastructure:**
- AWS account 123456789012
  - VPC vpc-prod-12345
  - EKS cluster prod-cluster
  - S3 buckets matching pattern acme-prod-*

### Out of Scope
- Production data modification
- Denial of service attacks
- Social engineering (separate engagement)
- Physical security testing
- Third-party hosted services (Cloudflare, Auth0)
- 203.0.113.50 (partner integration, written approval required)

## Rules of Engagement
- Stop testing immediately if production impact suspected
- Report critical findings within 4 hours of discovery
- No testing on weekends without explicit approval
- All testing traffic should use X-Test-Engagement: ACME-2025-03 header where possible

## Deliverables
- Draft report within 5 business days of testing completion
- Final report within 3 business days of feedback
- Executive summary (2 pages max)
- Technical findings with evidence
- Remediation guidance
- Retesting of critical/high findings included

Step 5: Define the rules of engagement#

When can testing happen?

  • Business hours only?
  • 24/7?
  • Specific maintenance windows?

What happens if something breaks?

  • Stop and report immediately?
  • Continue if impact is minimal?

How do you communicate?

  • Single point of contact?
  • Regular check-ins?
  • Emergency contact procedure?

Working With Pentesters#

Internal vs External#

Internal pentest team:

  • Knows your environment better
  • Faster to mobilise
  • Can test more frequently
  • May have blind spots from familiarity
  • May have conflicts of interest (testing their own work)

External pentest firm:

  • Fresh perspective
  • No organisational blind spots
  • Broader experience across industries
  • More expensive
  • Takes longer to understand environment

Best approach: both. Internal team for continuous testing, external for annual assessment. External validates internal isn’t missing things.

Red Flags in Pentest Proposals#

“We use proprietary AI-powered testing methodology”

No you don’t. You use standard tools and techniques. “AI-powered” is marketing. Ask what tools they actually use.

“We’ll test everything for $X”

You can’t test everything for any price. Scope is finite. If they claim otherwise, they’re lying or they don’t understand what they’re testing.

“We guarantee to find critical vulnerabilities”

No you don’t. You might find none. A clean report is possible. If they guarantee findings, they’ll manufacture them or find trivial issues and inflate severity.

“We’ll provide a pass/fail certification”

Penetration testing doesn’t pass or fail. It produces findings. You decide what to do with them. “Certification” is for compliance checkboxes, not security improvement.

What to Ask Potential Pentesters#

“Walk me through your methodology.”

They should describe something like:

  • Reconnaissance (what’s exposed, what technologies)
  • Vulnerability discovery (scanning, manual testing)
  • Exploitation (proving vulnerabilities are real)
  • Post-exploitation (what access enables)
  • Reporting (clear, actionable findings)

If they can’t explain it clearly, they don’t have a methodology. They have tools and hope.

“What’s your background?”

You want:

  • People who’ve done offensive security work
  • Certifications are fine but not required (OSCP, OSEE, GPEN demonstrate knowledge)
  • Experience in your industry (healthcare, finance, SaaS have different concerns)
  • Code background if testing applications

You don’t want:

  • Someone who took a course last month
  • Someone who only knows how to run tools
  • Someone who can’t explain what they’re doing

“Can I see a sample report?”

Redact client names, but show structure:

  • Executive summary (for management)
  • Technical details (for engineers)
  • Evidence (screenshots, commands, outputs)
  • Remediation guidance (how to fix)
  • Risk rating (critical/high/medium/low with justification)

If the sample report is vague, your report will be vague.

“What happens if you find a critical vulnerability on day 1?”

Good answer: “We notify you immediately, discuss whether to continue or pause, and adjust scope if needed.”

Bad answer: “We continue testing and include it in the final report.”

“Do you offer retesting?”

Good answer: “Yes, included in the engagement or as a follow-up at reduced rate.”

Bad answer: “That’s a separate engagement.”

What to Do With Findings#

Triage Immediately#

Pentest report arrives with 47 findings. Which ones matter?

Critical: Attacker can achieve their goal, or can get close with minimal effort. Fix now.

High: Attacker can gain significant access or information. Fix within days.

Medium: Attacker can gain limited access or information. Fix within weeks.

Low: Attacker can gain minimal access or information. Fix eventually.

But severity isn’t the only factor. Consider:

Exploitability: Is there a working exploit? Is it easy to use?

Business impact: What does this system do? What data does it hold?

Exposure: Is this internet-facing? Internal only? Air-gapped?

Compensating controls: Does something else partially mitigate this?

Pentest returns with these findings. Which do you fix first?

Sample Finding Structure#

A good pentest finding looks like this:

## Finding: SQL Injection in User Search

**Severity:** Critical
**CVSS:** 9.8 (AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H)

**Affected Asset:** https://app.acme.corp/users/search

**Description:**
The user search endpoint accepts a `name` parameter that is directly 
concatenated into a SQL query without sanitisation. An unauthenticated 
attacker can extract the entire user database.

**Proof of Concept:**
Request:
```http
POST /users/search HTTP/1.1
Host: app.acme.corp
Content-Type: application/x-www-form-urlencoded

name=admin' UNION SELECT id,email,password_hash,'4' FROM users--
```

Response contains user table data:
```json
[{"id":1,"name":"admin@acme.corp","email":"$2b$12$..."}]
```

**Impact:**
- Full database read access (customer PII, credentials)
- Potential database write access (modify/delete records)
- Compliance breach (GDPR, PCI-DSS)

**Remediation:**
1. Use parameterised queries for all database operations
2. Implement input validation whitelist (alphanumeric only)
3. Apply least-privilege database permissions
4. Add web application firewall rule as temporary mitigation

**References:**
- OWASP SQL Injection: https://owasp.org/www-community/attacks/SQL_Injection
- CWE-89: https://cwe.mitre.org/data/definitions/89.html

**Retest Required:** Yes

A medium finding on an internet-facing system might be higher priority than a critical finding on an isolated internal system.

Don’t Just Patch#

Pentester found SQL injection. You patch the specific injection point. Job done?

No. The finding tells you:

  • Input validation is missing or inadequate
  • The application wasn’t tested for this class of vulnerability
  • Developer training or code review missed this

Fix the finding. Then ask why it existed and fix the root cause.

Verify Fixes Work#

Fix deployed. How do you know it works?

Option 1: Retest by pentester. Best, but costs money and takes time.

Option 2: Internal verification. Reproduce the issue, verify the fix blocks it. Faster, cheaper, but might miss edge cases.

Option 3: Automated verification. Some vulnerabilities can be verified with automated tools. Quick but limited.

Do at least one. “We fixed it” without verification is hope, not security.

Track Over Time#

Findings this year vs last year:

  • Same vulnerabilities appearing? Root cause not addressed.
  • New vulnerability classes? Something changed in your stack or process.
  • Fewer findings? Maybe you’re improving. Maybe scope was smaller.

Track trends, not just individual findings.

Remediation Tracking Template#

Don’t let findings disappear into a PDF. Track them like any other work.

Summary section:

  • Total findings count by severity (Critical/High/Medium/Low)
  • Status counts (Fixed/In progress/Accepted risk)
  • Retest status (Retested/Verified fixed/Fix failed)

Findings table columns:

ColumnPurpose
IDUnique identifier (e.g., PENT-001) for cross-referencing
FindingShort description
SeverityCritical/High/Medium/Low
OwnerSpecific person responsible
StatusFixed/In progress/Accepted
DueTarget date
RetestPass/Fail/Pending
NotesAdditional context

Accepted risks table:

ColumnPurpose
IDLinks to finding
RationaleWhy accepted
Compensating ControlsWhat mitigates it
Review DateWhen to reassess

Retest results table:

ColumnPurpose
IDOriginal finding reference
Original FindingWhat was found
Fix AppliedWhat was changed
Retest DateWhen retested
ResultFixed/Failed
Follow-upNext action if failed

Example row:

| PENT-001 | SQL injection in search | Critical | Backend | Fixed | 2025-11-22 | Pass | Parameterised queries |

Track this in whatever system works - spreadsheet, Jira, Notion, GitHub issues. The format matters less than the tracking.

Key elements:

  • Unique ID for each finding (reference in all discussions)
  • Clear ownership (someone specific, not “the team”)
  • Due dates that are realistic
  • Retest status (don’t assume fix worked)
  • Accepted risks documented with rationale and review dates

Running Your Own Testing Program#

You don’t need to hire external pentesters for everything. Build internal capability.

Bug Bounty Programs#

Let anyone test your public-facing systems. Pay for valid findings.

Pros:

  • Continuous testing, not annual
  • Diverse attacker perspectives
  • Pay only for results

Cons:

  • Quality varies wildly
  • Can attract low-effort submissions
  • Requires triage resources
  • Public perception risk if not managed well

Best for: Mature organisations with public-facing applications and resources to triage submissions.

Internal Red Team#

Dedicated team that continuously tests your defences.

Activities:

  • Phishing campaigns (test detection and user awareness)
  • Network pentests (test segmentation and monitoring)
  • Application testing (test SDLC security)
  • Social engineering (test physical and human security)
  • Cloud configuration testing (test IaC and cloud security)

Pros:

  • Knows your environment
  • Can test continuously
  • Can coordinate with blue team for learning

Cons:

  • Expensive (dedicated headcount)
  • May develop blind spots
  • Can be seen as adversarial by other teams

Best for: Large organisations with security budget and mature security programs.

  • Small org: Annual external pentest + weekly vulnerability scanning
  • Medium org: Annual external + quarterly internal + continuous scanning
  • Large org: All of the above + red team + bug bounty

Purple Team Exercises#

Red team attacks. Blue team defends. They talk to each other.

Traditional: red team does something, blue team might detect it, report comes months later.

Purple team: red team says “we’re going to try technique X,” blue team watches for it, they discuss what worked and what didn’t.

Pros:

  • Faster learning
  • Both sides improve
  • Less adversarial

Cons:

  • Requires coordination
  • Less realistic than blind testing
  • Can become checkbox exercises

Best for: Organisations wanting to improve detection and response, not just find vulnerabilities.

Sample Purple Team Exercise#

Objective: Test detection of credential theft and lateral movement.

Red team action:

# Dump LSASS memory (Simulated attack)
mimikatz.exe "privilege::debug" "sekurlsa::logonpasswords" "exit"

# Use obtained credentials
runas /user:ACME\jsmith /netonly cmd.exe

# Lateral movement via WMI
wmic /node:10.0.0.50 /user:ACME\jsmith /password:... process call create "cmd.exe"

Blue team monitors for:

  • LSASS access by non-system processes
  • Unusual authentication patterns
  • WMI execution from non-admin workstations
  • Lateral movement to sensitive systems

Discussion points after exercise:

  1. Did SIEM alert on LSASS access?
  2. How long until someone investigated?
  3. Was the lateral movement detected?
  4. What visibility gaps exist?
  5. What detection rules need improvement?

Documentation:

# Purple Team Exercise: 2025-03-15

## Attack Emulated: Credential theft + lateral movement
## Detection Time: 4 minutes (LSASS), 12 minutes (lateral movement)
## Investigated: Yes (8 minutes after alert)
## Containment: Manual intervention required

## Findings:
1. LSASS alert fires but is buried in alert queue - increase priority
2. WMI lateral movement not detected for non-admin users - add rule
3. No alert when credential used from unusual workstation - add correlation

## Actions:
- [ ] Increase LSASS alert severity to critical
- [ ] Add detection rule for WMI from non-admin sources
- [ ] Create correlation rule for credential use from new workstation
- [ ] Schedule follow-up exercise for 2025-04-15

Continuous Vulnerability Scanning#

Not pentesting, but related. Automated scanning for known vulnerabilities.

Nuclei for CVE scanning:

nuclei -l urls.txt -t cves/ -t vulnerabilities/ -o nuclei-findings.txt

cat urls.txt
https://app.acme.corp
https://api.acme.corp
https://admin.acme.corp

Nikto for web server misconfigurations:

nikto -h https://app.acme.corp -output nikto-app.txt -Format txt
nikto -h https://api.acme.corp -output nikto-api.txt -Format txt

ZAP baseline scan for common issues:

docker run -t owasp/zap2docker-stable zap-baseline.py \
  -t https://app.acme.corp \
  -r zap-report.html

# More thorough scan (takes longer)
docker run -t owasp/zap2docker-stable zap-full-scan.py \
  -t https://app.acme.corp \
  -r zap-full-report.html

Scheduled scanning with cron:

# /etc/cron.d/vuln-scan
0 2 * * 1 security /opt/scans/run-weekly-scan.sh

#!/bin/bash
# /opt/scans/run-weekly-scan.sh

DATE=$(date +%Y%m%d)
SCAN_DIR="/var/scans/$DATE"
mkdir -p "$SCAN_DIR"

# Nuclei
cat /opt/scans/targets.txt | nuclei \
  -t /opt/nuclei-templates/cves/ \
  -t /opt/nuclei-templates/vulnerabilities/ \
  -o "$SCAN_DIR/nuclei.txt"

# Check for critical findings and alert
if grep -q "critical" "$SCAN_DIR/nuclei.txt"; then
  mail -s "[CRITICAL] Vulnerability scan found critical issues" \
    security@acme.corp < "$SCAN_DIR/nuclei.txt"
fi

# Nikto
while read url; do
  nikto -h "$url" -output "$SCAN_DIR/nikto-$(echo $url | sha256sum | cut -c1-8).txt"
done < /opt/scans/targets.txt

Run weekly. Triage results. Don’t let findings pile up.

This doesn’t replace pentesting. It handles the low-hanging fruit so pentesters can focus on interesting things.

Quick Checks Before You Pay for a Pentest#

Have you fixed the findings from the last pentest?

If no, another pentest won’t help. Fix what you already know about first.

Do you have a vulnerability management program?

If you’re not patching known vulnerabilities, a pentest will find them. Fix your patching process first.

Do you have basic security controls in place?

MFA, network segmentation, logging, patching. If not, pentest will find what you already know you’re missing.

Do you know what you’re worried about?

If you can’t articulate your top 3 security concerns, you’re not ready for a pentest. Figure out what matters first.

Do you have capacity to fix findings?

If pentest produces 50 findings and you can fix 5 per month, you’ll never catch up. Build remediation capacity first.

The Boring Truth#

Penetration testing is a tool, not a solution. A good pentest tells you what an attacker could do right now with a specific scope. That’s valuable. But it’s a snapshot, not a guarantee.

The organisations that get value from pentesting:

  1. Know what they’re worried about and scope accordingly
  2. Fix findings quickly and verify fixes work
  3. Address root causes, not just individual vulnerabilities
  4. Test continuously, not annually
  5. Build internal capability to supplement external testing

The organisations that don’t get value:

  1. Buy pentests for compliance checkboxes
  2. File reports without reading them
  3. Fix findings slowly or not at all
  4. Repeat the same pentest every year with same results
  5. Treat pentesting as a substitute for basic security hygiene

If you’re in the second group, save your money. A pentest won’t fix problems you’re not willing to fix yourself.

Ballpark Costs (UK/US, 2025)#

So you know what to budget. These are rough ranges - actual costs vary by provider, complexity, and location.

External network pentest (perimeter):

  • Small scope (5-10 IPs): £3,000-8,000 / $4,000-10,000
  • Medium scope (10-50 IPs): £8,000-20,000 / $10,000-25,000
  • Large scope (50+ IPs): £20,000-50,000+ / $25,000-60,000+

Web application pentest:

  • Simple app (10-20 endpoints): £5,000-12,000 / $6,000-15,000
  • Medium complexity: £12,000-25,000 / $15,000-30,000
  • Complex app (many features, auth flows): £25,000-50,000+ / $30,000-60,000+

Internal network pentest:

  • Small network (single site): £8,000-15,000 / $10,000-18,000
  • Medium network (multiple subnets): £15,000-30,000 / $18,000-35,000
  • Large enterprise: £30,000-75,000+ / $35,000-90,000+

Cloud infrastructure review:

  • Single cloud account: £5,000-12,000 / $6,000-15,000
  • Multi-account/multi-cloud: £12,000-30,000 / $15,000-35,000

Red team engagement (simulated attack):

  • Short (2 weeks): £20,000-40,000 / $25,000-50,000
  • Standard (4-6 weeks): £40,000-100,000 / $50,000-120,000
  • Extended (8+ weeks): £100,000+ / $120,000+

Bug bounty programs:

  • Platform fees (HackerOne, Bugcrowd): £15,000-50,000/year + bounties
  • Bounty budgets: Variable (typically £10,000-100,000/year for medium programs)

What affects price:

  • Scope size (obviously)
  • Complexity (custom protocols, unusual tech)
  • Timeline (rush jobs cost more)
  • Provider reputation (big names charge more)
  • Retesting included vs separate
  • Report quality (exec + technical vs technical only)

Warning signs:

  • Quote is 50% below market rate - corners being cut
  • Quote has no scope detail - they don’t know what they’re testing
  • Quote is time-based not scope-based - you’re paying for days, not results

Based on years of commissioning, receiving, and occasionally conducting penetration tests. Some were excellent and taught me things I didn’t know. Most were expensive confirmation of things I already knew. The difference wasn’t the pentester - it was how the engagement was scoped and what happened after the report arrived.
