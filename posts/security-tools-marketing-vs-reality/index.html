Vendor: Our AI-powered platform detects zero-day threats before they happen using quantum machine learning.

Sysadmin: Can I configure it?

Vendor: No, it’s automated intelligence.

Sysadmin: Can I see logs?

Vendor: No, proprietary algorithms.

Sysadmin: What exactly do I get?

Vendor: Peace of mind.

"Marketing departments discovered that 'AI-powered security' sells better than 'decent grep with sensible regex'. The rest is history."

Security vendor marketing has drifted so far from reality that products and datasheets might as well be different languages. Bought a product that promises “zero false positive threat hunting”? Got a glorified SIEM with slightly better alerting. Purchased “autonomous incident response”? Got some prewritten playbooks and a lot of configuration work.

This isn’t about shitting on all vendors. Some make genuinely good products. But the gap between what security teams think they’re buying and what they actually deploy is massive. And that’s before the sales team starts explaining how the “AI” part works.

The Marketing vs Reality Gap#

Let’s compare common vendor claims to what you actually get.

“AI-Powered Threat Detection”#

Marketing says:

Our platform uses advanced machine learning to detect unknown threats, zero-day exploits, and sophisticated attackers. No signature updates required. Set and forget.

Reality:

It’s anomaly detection on log data. Sometimes finds interesting stuff. Mostly finds legitimate admin activity that looks anomalous because your baseline is wrong. Requires constant tuning. Also needs signature updates for known threats because ML isn’t magic.

What actually works:

  • Decent anomaly detection (when properly tuned)
  • Basic ML on log patterns (surprisingly useful)
  • Still needs signatures for known threats
  • Still needs human review
  • Still produces false positives (just different ones)

What vendor doesn’t mention:

  • ML model needs training data from your environment (months of collection)
  • Baseline needs regular updating as your infrastructure changes
  • False positives from legitimate weird-but-safe activity
  • Requires security analysts to review alerts (not autonomous)

“Autonomous Incident Response”#

Marketing says:

When threats are detected, our platform automatically contains and remediates without human intervention. Security analysts sleep soundly while AI handles incidents.

Reality:

Some prewritten automation scripts. Maybe kill processes, isolate hosts, disable accounts. Most of the time requires manual approval because automated containment breaks production. Security analysts still wake up at 3am.

What actually works:

  • Basic automation (isolate suspicious host, disable compromised account)
  • Prebuilt response playbooks (same as writing your own scripts)
  • Some things can be automated safely (blocking IP, revoking API keys)
  • Most containment requires manual approval (breaking production is worse than breach)

What vendor doesn’t mention:

  • Automated containment breaks production regularly
  • Requires extensive testing before trusting automation
  • Most incidents require human judgment anyway
  • “Autonomous” is a generous term for “some prewritten scripts”

“Unified Security Platform”#

Marketing says:

Replace your entire security stack with our single platform. SIEM, EDR, cloud security, identity, compliance - all in one. Unified data, correlated insights, single pane of glass.

Reality:

It’s a decent SIEM with some integrations. Cloud security is separate module (extra cost). Identity integration requires expensive professional services. Compliance reporting exists but needs configuration. Single pane of glass mostly just filters and dashboards.

What actually works:

  • Good SIEM functionality (if you pay enough)
  • Some integrations out of box
  • Correlated logs are genuinely useful
  • One vendor to yell at when things break

What vendor doesn’t mention:

  • Each module costs extra (it’s not actually one price)
  • Deep integration with your existing stack requires professional services
  • Replacing everything is years of work
  • Best-of-breed tools usually beat unified mediocre ones

What Actually Works (Spoiler: Boring Stuff)#

After years of deploying, configuring, troubleshooting, and occasionally removing security tools, here’s what consistently delivers value.

1. Proper Monitoring and Alerting#

Not “AI threat detection.” Just knowing when something’s wrong.

What works:

  • Metric thresholds (CPU > 90% for 5 minutes, disk space < 10%)
  • Log parsing (failed login attempts, unusual admin activity)
  • Network flows (unexpected connections, data egress spikes)
  • Regular expression matching on logs (finds known attack patterns)

Why it works:

  • Understandable - you know what triggers alerts
  • Configurable - you can tune it without PhD
  • Debuggable - when it’s wrong, you can see why
  • Predictable - behaves consistently

The secret: Most vendors charge fortunes for “advanced analytics” that’s mostly decent regex and statistical thresholds. You can build 80% of this yourself with ELK, Prometheus, and some scripting. The remaining 20% might be worth paying for if you have the budget.

Concrete example (Prometheus): alert when SSH failures spike.

groups:
  - name: security_basics
    rules:
      - alert: SSHBruteForce
        expr: increase(ssh_failed_attempts_total[10m]) > 10
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "SSH brute force detected on {{ $labels.instance }}"
          description: "{{ $value }} failed attempts in 10 minutes (user={{ $labels.user }})"

If that metric doesn’t exist yet, that’s the point. You instrument the boring things you can act on.

2. Canary Tokens#

We’ve covered this before, but it bears repeating: canary tokens work.

What works:

  • DNS canaries planted in configs
  • Document-based canaries in sensitive shares
  • Credential honeytokens in code repos
  • File system canaries in unexpected locations

Why it works:

  • Near-zero false positives (canary should never be accessed)
  • Immediate detection (when attacker touches it, you know)
  • Simple to understand (no ML black box)
  • Cheap or free (canarytokens.org is literally free)

Vendor equivalent: “Deception technology platforms” charging tens of thousands per year for basically what you can do with a free service and some thought.

Concrete example (DNS canary in a config):

# /etc/app/config
backup_server=a1b2c3d4e5f6.canarytokens.com

If anything tries to resolve that hostname, you get an alert. Your job is to put it somewhere an attacker will trip over it while rummaging.

3. Vulnerability Scanning#

Not AI-predictive, just scanning. But scanning works.

What works:

  • Regular vulnerability scans (daily or weekly)
  • Container image scanning at build time
  • Dependency scanning in CI/CD
  • Regular penetration testing

Why it works:

  • Finds known vulnerabilities that attackers use
  • Integrates with patch management workflows
  • Provides prioritisation (critical vs medium vs low)
  • Understandable results (CVE with description)

What doesn’t work:

  • Relying solely on vulnerability scanning without patching
  • Scanning once per year (that’s compliance theater, not security)
  • Ignoring “medium” severity vulnerabilities (attackers use them)

Concrete example (containers):

trivy image your-image:tag
grype your-image:tag

Run it in CI. Fail the build on what you actually care about. Then fix the base image and rebuild.

4. Security Baselines#

Standardized, secure configurations. Boring but effective.

What works:

  • CIS Benchmarks for OS and applications
  • Cloud security baselines (AWS Security Hub, Azure Security Center)
  • Infrastructure as Code with security policies
  • Regular configuration audits

Why it works:

  • Reduces attack surface
  • Catches misconfigurations (actual common breach cause)
  • Automatable (no manual work)
  • Measurable (you can track compliance)

The vendor trap: “Continuous compliance monitoring platforms” that charge for scanning your infrastructure against standards you could run yourself with open source tools (OpenSCAP, Scout Suite, etc.).

Concrete example (Linux baseline):

lynis audit system

It’s not perfect. It’s still useful. The biggest value is it gives you a list of boring fixes you can apply this week.

5. Least Privilege Access#

Simple concept, hard to implement, massive impact.

What works:

  • Just-in-time access (grant access only when needed)
  • Role-based access control (minimal permissions per role)
  • MFA everywhere (no exceptions)
  • Regular access reviews (remove people who don’t need access anymore)

Why it works:

  • Limits blast radius of compromised credentials
  • Makes lateral movement harder
  • Doesn’t prevent compromise, limits damage
  • Works with any other security tool

The problem: It’s hard. Users complain. Management pushes back. Breaking things is common during implementation. But once it’s in place, it’s genuinely effective.

Concrete example (GitHub Actions permissions):

If you don’t set permissions, you often get a token with more access than the job needs.

name: CI
on: [push, pull_request]

permissions:
  contents: read

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: make test

This doesn’t solve everything. It stops “random workflow can write to the repo” as the default.

6. Incident Response Playbooks#

Not “autonomous AI response.” Just documented, tested procedures.

What works:

  • Clear runbooks for common incident types (compromised account, malware, data breach)
  • Contact information (who to call, when)
  • Escalation paths (when to escalate to management, legal, PR)
  • Regular drills (test the playbooks)

Why it works:

  • Less panic during actual incidents
  • Consistent response (not dependent on who’s on call)
  • Faster containment (you know what to do)
  • Post-incident learning (improve based on what went wrong)

Vendor twist: Some vendors sell “incident response platforms” that are basically document templates with some automation. Write your own playbooks, add some scripts, same result for less money.

What’s Usually Not Worth It#

Single-Vendor Unified Platforms#

The promise: one platform for everything. The reality: decent at most things, great at nothing.

Problems:

  • Each module costs extra (it’s not actually one price)
  • Integration with existing tools is painful
  • Vendor lock-in (switching is years of work)
  • When one part breaks, debugging across modules is nightmare

Better approach: Best-of-breed tools that integrate well. SIEM from one vendor, EDR from another, cloud native tools (CloudTrail, VPC Flow Logs, etc.). More complexity, but better functionality.

Over-Hyped AI Detection#

The promise: finds unknown threats. The reality: fancy anomaly detection with different false positives.

Problems:

  • Requires months of training data before it works
  • Still produces false positives (just different ones)
  • Insufficient explainability (why did this alert fire?)
  • Expensive (you’re paying for ML development costs)

Better approach: Start with solid monitoring (metrics, logs, alerts). Add ML-based anomaly detection if you have the budget and expertise to tune it properly. Don’t buy “AI” thinking it replaces security analysts.

Compliance-Only Tools#

The promise: checks all compliance boxes. The reality: generates reports, doesn’t improve security.

Problems:

  • Compliance ≠ security (you can be compliant and hacked)
  • Focuses on checkboxes, not actual risk reduction
  • Expensive for value delivered
  • Creates false sense of security (“we’re PCI compliant, we’re good”)

Better approach: Use compliance requirements as minimum baseline. Implement security that actually reduces risk, then document for compliance. Not the other way around.

A Practical Framework#

Here’s what I’d do if starting from scratch (or rebuilding security stack):

Tier 1: Foundation (Do This First)#

  1. Monitoring and alerting

    • Metrics: Prometheus/Grafana
    • Logs: ELK or Loki
    • Basic alerting on thresholds and log patterns
    • Cost: Free or minimal

  2. Security baselines

    • CIS Benchmarks for OS
    • Cloud security baselines (native tools, not third-party)
    • Infrastructure as Code with security policies
    • Cost: Time, not money

  3. Least privilege access

    • MFA everywhere
    • RBAC with minimal permissions
    • Regular access reviews
    • Cost: Political capital, not money

  4. Vulnerability scanning

    • Regular scans (Trivy, Clair)
    • CI/CD integration (scan images before deploy)
    • Dependency scanning (Snyk or similar)
    • Cost: Free or minimal

Tier 2: Detection (Do This After Foundation)#

  1. Canary tokens

    • DNS canaries in configs
    • Document canaries in sensitive shares
    • Credential honeytokens in code repos
    • Cost: Free (canarytokens.org)

  2. SIEM

    • Centralized log aggregation
    • Correlated alerts
    • Basic analytics
    • Cost: Elastic Cloud or similar (reasonable)

  3. Endpoint protection

    • EDR with detection capabilities
    • Behavioral analysis (if mature enough organization)
    • Cost: Per-endpoint licensing

Tier 3: Advanced (Do This If You Have Budget and Expertise)#

  1. Advanced threat detection

    • ML-based anomaly detection
    • User and entity behavior analytics (UEBA)
    • Threat intelligence feeds
    • Cost: Expensive

  2. Deception platforms

    • Beyond canary tokens
    • Honeypots, honey networks
    • Adaptive deception
    • Cost: Expensive

  3. Professional services

    • Security consulting
    • Penetration testing
    • Red teaming
    • Cost: Expensive but valuable

The Sales Pitch Reality Check#

Next time a vendor pitches you, ask these questions:

“How does it work?” If answer is “proprietary AI/ML,” that means “we won’t tell you.” Ask what data it uses, how it makes decisions, what you can configure.

“What’s the false positive rate?” Vendor will say “near-zero.” Push for actual numbers from real deployments. If they can’t provide, assume high.

“What happens when it’s wrong?” Can you see why it alerted? Can you tune it? Can you disable specific rules? If answer is “trust the AI,” walk away.

“What’s the rollback plan?” If tool breaks production or causes false positives, how fast can you revert? “We can turn it off” is not a plan. “We can disable this module in under 5 minutes” is a plan.

“Show me the logs/alerts.” Vendor should be able to show you example alerts from similar deployments. If they can’t, either they don’t have similar deployments or alerts are garbage.

“What’s the total cost?” Not just licensing. Implementation time, training, ongoing maintenance. Vendor sells you $50k/year license, then $200k in professional services. Make them include total cost in the pitch.

The Honest Truth#

Most security problems aren’t solved by buying fancier tools. They’re solved by:

  1. Implementing boring fundamentals (monitoring, baselines, least privilege)
  2. Configuring tools properly (not default settings, but tuned for your environment)
  3. Actually responding to alerts (not just tuning them down to zero)
  4. Regular testing (penetration testing, drills, tabletop exercises)
  5. Building security into development (shift left, not bolt on later)

AI-powered vendor platforms can help, but they’re force multipliers, not substitutes. Multiply your bad security practices, you get sophisticated failures. Multiply good practices, you get better security.

Start with the boring stuff. Get it right. Then consider the fancy stuff.

Because when the breach happens and management asks “why did our $200k AI platform miss this?”, you want to be able to say “because we didn’t have monitoring on this system” (fixable), not “because the AI black box didn’t think it was a threat” (not fixable).

Lessons from deploying, configuring, removing, and occasionally throwing away security tools across startups, banks, and everything between. Some good tools exist. Most are marketing wrapped around decent open source you could run yourself.
