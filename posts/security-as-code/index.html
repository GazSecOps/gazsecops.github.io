Security as Code: From Checklists to Automation

“Here’s the security checklist for the new release.”

“We’re deploying next week. When can we do the review?”

“The review takes two weeks minimum. You scheduled it last month.”

“…we didn’t.”

This is the problem with security checklists. They’re designed for a world where you deploy quarterly. In continuous deployment environments, they’re the wrong tool.

"Security checklists codify knowledge but don't scale. Security as Code codifies requirements into executable tests. Checklists tell you what to check. Code checks it for you, every time."

The evolution from checklists to code isn’t about abandoning security knowledge. It’s about making security verification scale with modern development practices.

What Checklists Actually Do#

Before we get into how to transform them, acknowledge what they did well.

Knowledge codification:

Security checklists captured institutional knowledge. Someone understood a requirement (like “databases must encrypt sensitive fields”) and wrote it down. New engineers didn’t have to rediscover that knowledge every time.

Consistent evaluation:

Different people reviewing the same system could reach different conclusions. Checklists created a common language and consistent criteria, reducing this variance.

Compliance mapping:

Well-designed checklists mapped requirements to regulatory frameworks. GDPR, PCI-DSS, SOC2 - these things map to checklist items in ways that make sense.

The problem wasn’t the checklist itself. The problem was how checklists were used.

The Limitation: Manual Verification#

Manual verification doesn’t scale.

Sysadmin: Did you check the password complexity policy in the new microservice?

Engineer: Yes, it requires 12 characters with special chars.

Sysadmin: How did you verify?

Engineer: I tried setting a password with 8 characters and it was rejected.

Sysadmin: Good. That’ll pass the review then.

This conversation happens constantly. Verification by attempting to break controls. It’s slow. It’s manual. It doesn’t happen on every deploy.

And it only catches things you think to test.

The Bridge: From Checklist to Code#

Security as Code isn’t about throwing away checklists. It’s about transforming checklist items into code that runs automatically.

Transformation Example 1: Password Policy#

Checklist item: “Applications must enforce password complexity: minimum 12 characters, uppercase, lowercase, numbers, special characters.”

Manual verification: Someone checks configuration settings or tries to set a weak password.

As code (policy definition):

resource "aws_iam_password_policy" "strict" {
  minimum_password_length        = 12
  require_lowercase_characters   = true
  require_uppercase_characters   = true
  require_numbers                = true
  require_symbols                = true
  allow_users_to_change_password = true
}

As code (verification):

control 'password-policy-1' do
  impact 0.7
  title 'Password Policy Configuration'  

  describe aws_iam_password_policy do
    it { should exist }
    its('minimum_password_length') { should be >= 12 }
    it { should require_uppercase_characters }
    it { should require_lowercase_characters }
    it { should require_numbers }
    it { should require_symbols }
  end
end

What changed:

  • Verification now runs automatically in CI/CD
  • Policy is defined as code (version controlled)
  • Test is repeatable and consistent

Transformation Example 2: Network Segmentation#

Checklist item: “Database servers must not be directly accessible from the public internet.”

Manual verification: Someone tries to connect from an external IP or checks firewall rules.

As code (policy definition):

resource "aws_security_group" "database" {
  name        = "database-sg"
  description = "Security group for database instances"
  vpc_id      = aws_vpc.main.id
  
  ingress {
    from_port   = 3306
    to_port     = 3306
    protocol    = "tcp"
    security_groups = [aws_security_group.application.id]
  }
  
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

As code (verification):

package terraform.analysis

import input.plan as tfplan

deny[msg] {
  sg = tfplan.resource_changes[_]
  sg.type == "aws_security_group"
  sg.change.after.ingress[_].cidr_blocks[_] == "0.0.0.0/0"
  sg.change.after.ingress[_].to_port == 3306
  
  msg = sprintf(
    "Security group '%s' allows public access to database port 3306",
    [sg.change.after.name]
  )
}

What changed:

  • Network isolation defined declaratively
  • Policy verification runs automatically on infrastructure changes
  • Prevents drift (can’t manually open port after policy is set)

Transformation Example 3: Encryption at Rest#

Checklist item: “All sensitive data must be encrypted at rest using approved algorithms.”

Manual verification: Someone checks database configuration, storage settings, documentation.

As code (policy definition):

resource "aws_s3_bucket" "data" {
  bucket = "sensitive-data-bucket"
  
  server_side_encryption_configuration {
    rule {
      apply_server_side_encryption_by_default {
        sse_algorithm = "AES256"
      }
    }
  }
}

As code (verification):

package terraform.analysis

import input.plan as tfplan

# Require default encryption on S3 buckets
deny[msg] {
  rc := tfplan.resource_changes[_]
  rc.type == "aws_s3_bucket"
  not rc.change.after.server_side_encryption_configuration

  msg := sprintf(
    "S3 bucket '%s' has no default encryption",
    [rc.change.after.bucket]
  )
}

# If you require KMS (not AES256), enforce that too
deny[msg] {
  rc := tfplan.resource_changes[_]
  rc.type == "aws_s3_bucket"
  sse := rc.change.after.server_side_encryption_configuration.rule[_].apply_server_side_encryption_by_default
  sse.sse_algorithm != "aws:kms"

  msg := sprintf(
    "S3 bucket '%s' is not using KMS encryption",
    [rc.change.after.bucket]
  )
}

What changed:

  • Encryption enforced by infrastructure definition
  • Encryption is verified in tests before code ships
  • Configuration is version controlled and auditable

The Organizational Shift#

This transformation requires more than changing tools. It changes how teams work together.

From Gatekeeper to Enabler#

Traditional security: Teams submit code for review. Security team reviews it (slowly). Security team gatekeeps deployment.

Security as Code: Security team provides tests and policies. Dev team runs them in CI/CD. Security team reviews test results. Deployment happens when tests pass.

The difference: Security isn’t blocking deployments. Security is enabling faster deployments by making verification automatic.

Before:

  • Security review takes 2 weeks
  • Changes sit waiting
  • Pressure to skip review
  • Security team seen as obstruction

After:

  • Tests run in minutes
  • Results are immediate
  • Security team sees test output, reviews failures
  • Deployment continues if tests pass

From Auditor to Engineer#

Traditional security teams function as auditors. They review systems periodically. They issue findings. They check if findings were fixed.

Security as Code transforms security teams into engineers who build tools. They create tests. They create policies. They integrate these into CI/CD.

Management: Why did we fail the last audit? Same finding as previous year.

Sysadmin: Because it was a checklist item. Someone forgot to check it.

Management: Can we make it so they can’t forget?

Sysadmin: Yes. Make it a test that runs on every deploy.

Management: What if someone tries to bypass it?

Sysadmin: CI/CD blocks deployment if test fails. They’d have to ask you to override.

Practical Implementation#

Don’t try to convert entire security program to code at once. That’s how big transformation projects fail. Start small.

Stage 1: Digitize Checklists#

Convert existing checklists into structured formats with metadata. Not just text documents.

Format:

  • Control name
  • Requirement description
  • Compliance mapping (PCI-DSS 8.2.4, GDPR Article 32, etc.)
  • Risk rating (high, medium, low)
  • Implementation guidance

This creates foundation for automation.

Here’s a simple template that works well enough to start:

id: SEC-001
name: Database not publicly accessible
risk: high
compliance:
  - pci_dss: "1.3.1"
  - gdpr: "Article 32"
owner: platform-team
scope:
  - terraform
applies_to:
  - aws_db_instance
requirement: "Database must not have a public endpoint"
implementation_guidance: "Put DB in private subnets; restrict SG ingress to app tier"
verification:
  type: policy
  tool: conftest
  evidence: "OPA policy passes in CI; terraform plan JSON attached"
exemptions:
  allowed: false

Stage 2: Identify Automatable Items#

Not everything can be tested automatically. Some things require human judgment.

Automatable:

  • Password policies (can be verified via code)
  • Network rules (can be tested via infrastructure-as-code tools)
  • Encryption configuration (can be verified)
  • SSL/TLS certificates (can be checked)

Not easily automatable:

  • Security architecture reviews
  • Threat modeling sessions
  • Incident response quality assessments

Focus automation on what can be tested automatically. Keep human review for what can’t.

Stage 3: Build Tooling#

Choose tools that fit your existing workflows.

If you use Terraform:

  • Write tests in Terraform itself (using check blocks) or Open Policy Agent (OPA)
  • Run tests in CI/CD

If you use Kubernetes:

  • Kyverno for policy enforcement
  • OPA Gatekeeper for admission control

If you use CI/CD:

  • GitHub Actions or GitLab CI
  • Custom test runners or commercial SAST tools

Don’t introduce a whole new toolchain just for security. Use what you already have.

Stage 4: Integrate with Deployment Pipeline#

Security tests run in same pipeline as other tests. They block deployment if they fail.

# Example GitHub Actions workflow
name: Security Tests
on: [pull_request, push]

jobs:
  security-scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      # Run InSpec tests
      - name: Run InSpec
        run: |
          inspec exec test/integration/default
      
      # Run Terraform security scan
      - name: Run tfsec
        run: |
          tfsec .
      
      # Check dependencies
      - name: Run npm audit
        run: |
          npm audit --audit-level=moderate
      
      # Run SAST scan
      - name: Run Snyk
        uses: snyk/actions/node@master
        with:
          args: --severity-threshold=high

Security becomes like linting. It happens automatically, fails builds if issues are found, and requires explicit overrides.

Stage 5: Shift Left#

Security tests happen earlier in development cycle. Not just before deployment.

Development:

  • SAST scanning on every commit
  • Dependency scanning in CI
  • Security-focused unit tests

Staging:

  • DAST scanning on staging deployments
  • Penetration testing on staging environment
  • Security integration tests

Production:

  • Runtime security monitoring
  • Cloud security posture management (CSPM)
  • Automated compliance verification

The goal: Catch issues before they reach production.

The Skills Gap#

Security teams lack software engineering skills. Development teams lack security expertise. Both need training.

For security professionals:

  • Learn automation tools (InSpec, OPA, Checkov)
  • Learn CI/CD integration (GitHub Actions, GitLab CI, ArgoCD)
  • Learn programming basics (Python, Go, shell scripts) to write tests
  • Focus on building tools, not just reviewing systems

For developers:

  • Learn security fundamentals (OWASP Top 10, CWE, CVEs)
  • Learn secure coding practices (input validation, output encoding, authentication)
  • Learn security testing tools (how to interpret results, how to fix findings)
  • Treat security as quality, not compliance

Cross-training: Security engineers pair with developers on implementing security controls. Developers pair with security engineers on threat modeling. Both learn from each other.

Most effective organizations I’ve seen have security engineers embedded in development teams, not sitting in a separate security department reviewing code.

Common Anti-Patterns#

Anti-pattern 1: Security testing by security team only

Only security team should run security tests. Dev team can’t understand results. Can’t fix findings quickly.

Better: Security team creates tests. Dev team runs tests daily. Dev team can fix issues immediately.

Anti-pattern 2: Security gates without feedback

Security tests fail deployment. No explanation of why. Dev team disables tests to ship.

Better: Security tests produce actionable output. Failures include links to remediation guidance. CI/CD provides mechanism for documented exemptions.

Anti-pattern 3: Point-in-time compliance

Run security tests annually for audit. Ignore security rest of year.

Better: Security tests run continuously. Compliance is continuous state, not annual event.

Anti-pattern 4: Separation without collaboration

Security team develops tests in isolation. Dev team doesn’t understand requirements.

Better: Security and dev teams collaborate on defining tests. Dev team provides context about system architecture. Security team provides guidance on security requirements.

What Good Looks Like#

Effective Security as Code implementations share characteristics:

Continuous verification: Security checks run on every commit, every pull request, every deploy. Not periodically.

Fast feedback: Developers get security feedback in minutes, not weeks. Tests fail builds, not require manual review cycles.

Clear policies: Security requirements are expressed as code, not documents. Policy is the source of truth.

Blocking: Security tests that fail block deployment. No exceptions without documented override process.

Transparent: Test results are visible to all stakeholders. Everyone can see what passed and what failed.

Developer-friendly: Tests are easy to run locally. Dev team can fix issues before creating pull request.

Security-appropriate: Tests catch real security issues, not just checklists. Focus on vulnerabilities, not compliance artifacts.

Measurement#

How do you know if transformation is working?

Metrics to track:

  • Time from security finding to remediation (days, not weeks)
  • Percentage of security checks automated
  • Deployment frequency blocked by security tests
  • Vulnerability remediation rate (what percent of findings are fixed before next release)
  • Mean time to detect production security issues (MTTD)

Targets:

  • 90% of security controls automated
  • Security findings fixed within 7 days
  • Less than 5% of deployments blocked by security
  • Mean time to detect security issues in production < 24 hours

These are realistic targets for organizations making this transition.

The Boring Truth#

Security as Code isn’t a product you buy. It’s not a framework you implement. It’s a way of working.

The transformation happens incrementally. Not in one big rewrite project, but in small steps that each improve how security is done.

Start with the most automatable items. The ones that are most frustrating in manual review. Password policies. Network rules. Encryption configuration.

Automate those first. Show value quickly. Then expand to more complex areas.

The goal isn’t 100% automation. Some security review requires human judgment. The goal is 80% automation, with human review focused on the 20% that requires it.

Most organizations fail this transformation by trying to do everything at once. Security team tries to automate everything. Dev team resists changes. Management demands unrealistic timelines.

Successful transformations pick one thing, do it well, prove value, then pick the next thing.

Your security checklists captured valuable knowledge. Transform that knowledge into code. Don’t lose what worked. Make it scale.
